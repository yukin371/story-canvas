# Classic Structure Techniques

这是 `story-harness-writing` 的按需结构技法层。

读取时机：

1. 用户要求避免平铺直叙、增强悬念、反转、双线、倒叙、以小见大或结尾余味。
2. 需要在起稿前选择叙事策略，而不是只补 `direction / beats / scenePlans`。
3. 需要把传统写作法翻译成 Story Canvas 的 CLI 前置约束和复检点。

不要把本文件当成协议 schema。真正的项目真相仍是 `project.yaml`、`PRD.md`、`outline.yaml`、`detailed_outlines.yaml` 和 CLI 输出。

## 0. 层级边界

1. 三幕式、五幕式、英雄之旅、Save the Cat：偏 `act` / 卷级 / 全书级结构模板。
2. 下列十种技法：偏章节、单元案、短篇或局部叙事策略。
3. `scenePlans`：章节内场景 / 场次 / 功能段，负责局部目标、阻力和变化。

不要把 `scene` 翻译成传统结构里的“幕”。需要说“幕”时，优先写 `act` 或“结构幕”。

## 1. Skill + CLI 使用方式

写前先选一种主技法，最多再选一种辅技法。不要一章同时塞满多种技法。

把技法翻译成 CLI 可检查字段：

1. `PRD.md`: 写清本卷采用的主结构法、卷问题、读者钩子、承接点和交付点。
2. `storyContract.corePromises`: 写主承诺，避免技法变成纯花活。
3. `paceContract`: 写本卷使用的结构节奏，例如“三章首卷：悬念切入 -> 追查解码 -> 局部归档”。
4. chapter `direction`: 写本章技法如何服务当前章。
5. `beats`: 写不可删除的推进节点。
6. `scenePlans`: 写每个场景的目标、阻力、变化和尾部余波。

强制复检：

1. `outline check` 确认方向、beats、scenePlans 齐备。
2. `review chapter` 看章节是否有真实推进、承接和风格风险。
3. `review scene` 看局部场景是否只是解释，还是有目标、阻力、变化。
4. `review preflight --volume-id` 看技法是否服务卷级闭环，而不是只制造悬念。
5. `review volume-self` 归因：问题属于生成漏做、自审漏看还是工具没检出。

## 2. 十种技法

### 2.1 切入悬念，倒叙事件

别名：横切悬念、悬念切入、倒叙追因。

做法：

1. 首段或开头场景先给一个笼罩全篇的疑团。
2. 疑团必须具体可感，不是抽象口号。
3. 随后倒叙、补叙或追查事件来源，让读者为了答案继续读。

适用：

1. 悬疑、都市异闻、单元案、人物秘密揭示。
2. 首章需要快速建立追读理由。

CLI 落点：

1. `direction`: 写“开头悬念是什么，后文追查什么原因”。
2. `beats`: 至少包含“悬念呈现 / 证据反常 / 倒叙或追查入口”。
3. `scenePlans`: 第一场景必须有可见疑点，后续场景逐步给证据，不要只拖延答案。

风险：

1. 只设谜不回收，会变成空钩子。
2. 悬念必须在人物可感范围内，避免 POV 越界。
3. 倒叙不能长到打断当前行动压力。

### 2.2 意料之外，情理之中

做法：

1. 让情节结果超出读者和角色预期。
2. 回看时必须符合人物性格、动机、世界规则和生活逻辑。

适用：

1. 反转结尾、礼物错位、身份揭示、规则二次解释。

CLI 落点：

1. `beats`: 先埋性格 / 规则 / 物件证据，再交出反转。
2. `review chapter`: 重点看反转是否有前置证据。
3. `review scene`: 重点看反转场景是否靠行动和物件兑现，而不是作者解释。

风险：

1. 没有前置证据的反转是硬拧。
2. 只追求惊讶会破坏人物连续性。

### 2.3 淡化情节，形散神聚

做法：

1. 外层情节可以淡，但内在线索必须统一。
2. 靠反复意象、情绪流向、人物关系微变和主题回声聚拢。

适用：

1. 文学性短篇、日常流、成长回忆、低冲突情感单元。

CLI 落点：

1. `storyContract.corePromises`: 写清情绪承诺，不用强行制造大事件。
2. `scenePlans.summary`: 每个场景必须写“情绪或关系变化”，不能只写日常动作。
3. `volume-self`: 检查是否有内在精神线，而不是散到没有交付。

风险：

1. 低冲突不等于无变化。
2. 工具可能低估这种写法的张力，需要在自审里写明内在线索证据。

### 2.4 一箭双雕，一点两面

做法：

1. 用一个道具、事件、制度或选择同时照出两方人物。
2. 同一对象既推动情节，又揭露关系、阶层、欲望或价值差异。

适用：

1. 阶层冲突、职场权力、悬疑证物、关系试探。

CLI 落点：

1. `beats`: 写“道具出现 / 两方反应 / 道具造成选择或暴露”。
2. `scenePlans`: 明确这个道具如何分别影响双方。
3. `review scene`: 看道具是否只做装饰，还是改变局面。

风险：

1. 道具不能只承担解释功能。
2. 双面必须来自角色行动，不是作者旁白总结。

### 2.5 偶然中必然，必然中偶然

做法：

1. 表面由偶然事件启动。
2. 深层由人物欲望、社会处境、规则压力或长期选择造成必然结果。
3. 结尾可再揭示一个改变理解的偶然事实。

适用：

1. 命运感故事、社会讽刺、误会链、代价链。

CLI 落点：

1. `beats`: 写清偶然触发、必然压力、代价链、最终揭示。
2. `chapter review`: 看事件链是否因果成立。
3. `volume-self`: 检查读者回看时是否能理解“为什么只能这样发生”。

风险：

1. 偶然太多会像巧合堆砌。
2. 必然只靠作者解释，会失去戏剧力。

### 2.6 银丝串珠，数点一线

做法：

1. 表面用多个片段、视角、人物或荒诞侧面分散叙述。
2. 内部用一个主题、规则、意象或后果把片段串成一线。

适用：

1. 群像、末世、荒诞、意识流、单元碎片拼图。

CLI 落点：

1. `PRD.md`: 明确银丝是什么，如共同主题、同一灾难后果、同一物件。
2. `beats`: 每个片段必须贡献一个不同侧面。
3. `volume-self`: 检查“形散”是否真的“神聚”。

风险：

1. 片段之间没有内在回声，会变成素材堆。
2. 工具可能误判章间承接弱，需要用 `closureAssessment` 写明统一线索。

### 2.7 明线暗线，双环连套

做法：

1. 明线负责可见行动和主要故事。
2. 暗线负责隐藏人物、历史原因、制度真相或主题反讽。
3. 两线在关键节点交叉，让结尾主题自然显形。

适用：

1. 悬疑、革命 / 历史题材、双主角、主线 + 单元案。

CLI 落点：

1. `threads.yaml`: 明线和暗线各自建 thread。
2. `beats`: 标注本章推进明线还是暗线。
3. `foreshadowing.yaml`: 暗线证据优先入账，避免丢。
4. `volume-self`: 检查两线是否有交叉点，而不是平行不相干。

风险：

1. 暗线不能只藏到最后才出现。
2. 明线不能沦为给暗线服务的空壳。

### 2.8 欲扬先抑 / 欲抑先扬

做法：

1. 要突出人物能力、品格或价值，先给压低印象，再用行动翻转。
2. 要揭露虚伪或失败，先给表面威风，再让事实击穿。

适用：

1. 人物登场、反派压迫、成长打脸、身份误认。

CLI 落点：

1. `direction`: 写清本章先压什么、后扬什么，或先扬什么、后抑什么。
2. `beats`: 必须有“初始误判 / 行动证据 / 认知翻转”。
3. `review chapter`: 看翻转是否靠行动，而不是作者宣布。

风险：

1. 先抑过头会让人物失去可信度。
2. 先扬的对象必须真的有压迫感，否则后抑无力。

### 2.9 盆中藏月，以小见大

做法：

1. 用小物件、小场景、小生活困境承载更大社会、制度或人生主题。
2. 不靠大事件，而靠具体选择和处境让读者自己看见大问题。

适用：

1. 现实题材、社会派、生活流、短篇。

CLI 落点：

1. `storyContract.corePromises`: 写“小事背后的大问题”。
2. `scenePlans`: 场景必须具体，物件必须能造成选择。
3. `volume-self`: 检查主题是否从故事里长出来，而不是结尾喊口号。

风险：

1. 主题太直白会变成议论文。
2. 小物件必须进入人物处境，不能只做象征。

### 2.10 余音绕梁，三日不绝

做法：

1. 结尾不只结束事件，还留下情绪、意象、代价或新理解。
2. 最后一个动作、物件或句子应回响全篇最核心的变化。

适用：

1. 卷尾、短篇结尾、单元案收束、人物关系转折。

CLI 落点：

1. `beats`: 最后一项写“收束动作 + 余味对象”。
2. `scenePlans`: 末场景必须回答“收了什么，留下什么”。
3. `volume-self`: 检查结尾是局部 closure，而不是只抬新谜题。

风险：

1. 只留悬念不交付，会削弱闭环。
2. 余味不能靠抽象总结，优先落到物件、动作、沉默或关系边界。

## 3. 技法选择规则

起稿前只保留短 style/structure brief：

```text
主技法：
辅技法：
卷级 act 模板：
本章悬念 / 反转 / 道具 / 双线：
必须兑现：
禁止写成：
CLI 落点：
```

不要把上面字段写进正文。

常见组合：

1. 都市超凡悬疑：切入悬念 + 明线暗线。
2. 单元案：一箭双雕 + 偶然中必然。
3. 文学短篇：淡化情节 + 盆中藏月。
4. 群像荒诞：银丝串珠 + 余音绕梁。
5. 爽文人物登场：欲扬先抑 / 欲抑先扬 + 意料之外情理之中。

## 4. 前置约束与强制约束

前置约束适合写入 skill / PRD / outline：

1. 本卷使用哪种 act 模板。
2. 本章使用哪种主技法。
3. 读者开头看到的疑问是什么。
4. 本章必须交付的答案或状态变化是什么。

强制约束适合交给 CLI gate：

1. `outline check`: 没有 direction / beats / scenePlans 不开写。
2. `review chapter`: 没有推进、承接、冲突或风格失真就返修。
3. `review scene`: 场景没有目标、阻力、变化就返修。
4. `review preflight`: 卷级结构、实体、世界、伏笔、scene review 聚合不过线就不进卷审。
5. `review volume-self`: 没有闭环判断、缺陷归因、editor pass 和 accepted risks 不进人工审查。

设计原则：

1. skill 负责选择技法和把技法翻译成写作 brief。
2. CLI 负责验证字段、证据、评分和 gate。
3. 不用 CLI 强行判断“这是第几种技法”；优先判断它是否真的产生了悬念、反转、双线、交付和余味。
