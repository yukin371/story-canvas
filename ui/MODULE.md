# UI 模块说明

> 最后更新: 2026-04-28
> 状态: 当前有效模块文档

## 1. 模块职责

- 提供 `Story Canvas` 的早期本地视觉壳
- 消费现有项目协议与本地 API，只做浏览、预览和操作编排
- 降低章节审查、插画测试、结果确认、配置校验与历史复用的人肉成本
- 当前工作台 shell 采用窄 activity bar + VS Code 风格 explorer 侧栏，负责项目切换与工作区快速跳转
- App shell 负责活动栏与状态栏，重工作区视图应优先通过异步组件装配，避免把整块工作台逻辑压回主入口
- App shell 当前直接负责 review / illustration / settings 的页面切换；不要再并行维护一套未接入真实入口的旧路由页
- TDesign 本地适配层若出现共享 chunk 膨胀，应优先拆到组件级 wrapper，而不是继续扩大 `forms` / `display` 这类粗粒度 barrel
- TDesign 的 JS 组件可继续走本地 wrapper 按需拆分，但基础样式入口应保持单点注入，避免开发态或异步视图切换时出现“组件逻辑在、样式缺失”的半失效状态
- 审查工作区首屏若只需要项目切换、导入或创建这类轻交互，应优先使用原生控件或极轻壳实现，避免把 `select` / `input` / `button` 这类 TDesign 表单运行时重新压回 review 路径
- 插画工作区中的重型辅助界面（如模板管理面板）应尽量按需异步加载，不要默认压进审查模式或常驻首批 chunk
- `WorkbenchView` 若继续承载 review / illustration 双模式，应优先把模式主体拆成异步子视图，而不是把两套模板长期堆在同一 chunk
- 当前 `WorkbenchView` 已退回模式装配壳；`ReviewWorkbenchPane`、`IllustrationWorkbenchPane` 与模板编辑抽屉应继续各自维持独立 chunk
- `WorkbenchView` 自身不应再持有整套 review / illustration 业务状态；模式内的状态、watcher 与 API 装配应下沉到各自异步容器，避免审查路径被插画逻辑拖入共享包
- 项目入口默认自动扫描仓库 `projects/` 目录；工作台私有注册文件只额外记录“最近项目”和“外部导入项目”
- 工作台内允许从 UI 直接创建最小项目骨架，但项目结构仍必须与 CLI `project init` 协议兼容
- 外部项目导入若需要目录选择，必须通过本地 UI API 的原生目录选择桥接完成，不能伪装成浏览器上传根目录
- 插画工作区必须暴露真实可操作输入，包括目标、用途 use case、正面提示词、负面提示词、模板、修饰词、批量数量、源图和 mask；未绑定项目时仍应允许自由生图，但结果只落在工作台私有缓存，不写回项目协议
- 插画工作区的历史区允许同时查看“当前项目记录”和“自由模式缓存记录”，但这两类历史都必须来自后端真实协议/沙箱摘要；前端只能做聚合展示，不能补写一份历史状态
- 插画工作区中的“纯绘图 / 文字设计”必须是同一条插画请求上的正交参数：允许显式关闭文字版式，也允许补标题 / 副标题 / 简介 / 字体气质提示；这组数据仍应随同插画请求回传后端，而不是在前端独立持久化
- 插画工作区中的提示词系统应保持轻量模板编排器定位：模板与修饰词由后端提供，前端只负责按 `source` 区分系统/用户模板、允许补充文本并回传生成请求
- 用户自定义模板若由前端创建或编辑，必须回传本地 API 落到项目或工作台自身的 `prompts/illustration-packs/`，不能只停在前端内存态；模板包文件名允许使用中文，只需清理非法路径字符
- 审查工作区展示章节时，正文内容与审查摘要必须分开传输和渲染；不得再把 review summary 伪装成章节正文
- 审查工作区的角色卡、世界设定、卷结构与审查包浏览必须来自 `ProjectSummary` 的只读聚合；前端可以切换阅读面板，但不能回写这些协议事实
- 当前打开项目属于工作台私有状态，应通过本地 API 持久化到项目 registry；刷新恢复只能读取该 registry，不能写入项目协议

## 3. 不变量

- UI 不是新的真相源；项目真实状态仍以 CLI + 协议文件为准
- UI 只能消费已有协议与 API 输出，不能在前端发明持久化字段
- 现阶段 UI 只作为本地工作台，不改变 `43187` 前端 / `43188` API 联调约定
- 若后端尚未暴露某类数据，前端只能做只读整形或明确标注为待接入，不能伪装成真实持久化结果
- 插画工作台的真实生成、历史记录和表单回填必须继续复用本地 API 与协议字段，不能在前端单独维护任务状态
- 插画历史里的目标显示可优先使用后端补出的 `targetName` 等只读字段，但不能在前端自行推断并写回新的目标元数据
- 系统模板与用户模板的来源区分必须来自后端返回的 pack 元数据；前端不得自行推断目录结构或生成新的模板真相源
- 本地工作台级 secret 不进入项目协议；敏感配置仅允许通过本地 API 的工作台私有配置持久化
- 工作台私有 provider 配置可包含默认 API key 与默认 Base URL，但只能作为本地回退，不得伪装成项目协议字段
- 若浏览器侧需要上传参考图 / mask，文件桥接与资产预览必须仍由本地 UI API 承担；前端只负责 FormData 提交与预览，不直接改写协议 owner
- 结果图导出与“打开文件夹”这类本机文件操作也必须继续走本地 UI API 桥接；前端只负责发起动作，不直接假设浏览器拥有本地文件系统权限
- 多 provider 凭证、脱敏显示与优先级 fallback 也属于工作台私有配置职责，不得下沉到项目协议或前端本地假状态
- UI 创建项目时不得发明第二套项目布局；最小骨架、首章 stub 和状态文件仍需兼容 CLI 已有协议
- `projects/` 自动发现是默认项目源；最近项目与外部导入路径只允许落在工作台私有注册文件中，不得写回项目协议
- `project-registry.json` 可以记录 `activeRoot` 作为本地工作台恢复状态；退出项目必须清空该字段，避免刷新后误恢复旧项目
- TDesign 组件不得再通过全量插件一次性灌入根应用；应优先通过本地分域入口按需导入，维持可控 chunk 边界

## 4. 文档同步触发条件

- UI 页面职责变化
- 新增或删除主要工作台视图
- UI 与本地 API 的契约变化
- 本地联调、构建或运行方式变化
- `ui/design.md` 的设计规则变化
