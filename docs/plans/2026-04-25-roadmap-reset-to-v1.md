# 路线图重置到 v1 版本线

> 日期: 2026-04-25
> 状态: 已采纳，待体现在 `docs/roadmap.md`
> 范围: 文档治理，不涉及代码实现

## 1. 背景

现有 `docs/roadmap.md` 仍停留在 `v0.2 已完成` 的收尾状态，已经不能准确表达接下来一段时间的执行入口。

用户已明确给出新的产品方向：

- `1.0`：整体发布版本，要求具备完整、有效的故事管理能力，项目可像代码仓库一样管理，同时具备简单生图管理能力与工作流能力；由于后续会增加 Web UI，需要开始考虑仓库改名
- `1.1`：简易 Web UI，用于阅读、审查与简单修改产出物
- `1.2`：丰富模板、提示词管理、工作流自由编排，提供默认模板与参考

## 2. 重置原则

### 2.1 `docs/roadmap.md` 继续作为唯一执行入口

不再把 `v0.2` 的完成记录保留为执行入口，只保留必要的历史背景。

### 2.2 版本线从“功能补洞”切换为“产品发布”

新的路线图不再按 P0-P3 缺陷/补功能方式组织，而改为：

- `1.0` 发布验收
- `1.1` Web UI 最小闭环
- `1.2` 模板与编排层

### 2.3 `1.0` 的重点是协议与闭环，而不是堆功能

`1.0` 必须回答：

- 故事项目是否已经具备稳定协议
- 项目是否真的能像代码仓库一样被管理
- 核心 CLI 闭环是否可验证、可迁移、可审查

## 3. 版本定义

### 3.1 v1.0

定位：

- 第一个面向外部发布的“可用产品版本”

重点：

- 完整故事管理能力
- 简单生图管理能力
- 工作流能力
- 仓库/产品命名评估

不应承诺：

- 完整 Web UI
- 高自由度编排系统
- 重型模板市场

### 3.2 v1.1

定位：

- 为现有协议提供轻量 Web UI 外壳

重点：

- 阅读
- 审查
- 简单修改结构化产物

约束：

- UI 不得成为新的真相源，仍以文件协议为准

### 3.3 v1.2

定位：

- 在稳定协议与 UI 基础上，提升复用性与可配置性

重点：

- 模板体系
- prompt pack / 提示词管理
- workflow 自由编排
- 默认参考项目与模板库

## 4. 进入 `docs/roadmap.md` 的建议结构

建议改为：

1. 当前发布目标：`v1.0`
2. `v1.0` 验收范围
3. `v1.0` Active Tracks
4. 当前阻塞
5. `v1.1` 展望
6. `v1.2` 展望

## 5. v1.0 建议验收口径

- 项目能以稳定文件协议管理，而不是依赖隐式状态
- 核心故事管理域完整：项目定位、世界/角色/伏笔/线程/时间线/评审/投影/工作流
- CLI 闭环稳定：规划、写作、分析、评审、投影、上下文刷新、导出
- 简单插图资产管理可用：prompt、生成记录、资产落盘、引用关系
- 至少有一个长篇样例与一个短篇样例可证明真实闭环
- 命名与仓库边界有明确结论：是否从 `story-harness-cli` 收敛到更通用的产品名

## 6. 风险提醒

- 若 `1.0` 同时塞入 Web UI，会稀释发布标准
- 若 `1.0` 不先收敛协议，`1.1` 的 UI 很容易反过来绑架协议
- 若 `1.2` 先做编排自由度而默认模板不稳，会得到高复杂度、低可用系统

## 7. 结论

本次路线图重置的核心不是改数字，而是把仓库从“持续补功能的 CLI 项目”切换到“面向发布的故事工程产品”。
