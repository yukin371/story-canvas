## 1. 产品定义

| 字段 | 内容 |
|------|------|
| 需求名称 | 写作质量与工作流收敛 |
| 产品归属 | Story Canvas `v1.0` 核心故事工作流 |
| 核心对象 | AI 写作代理、作者、审查者 |
| 核心目标 | 让长篇小说生成从“能连续出稿”收敛成“能稳定交付卷级小故事闭环” |
| 主要范围 | 写作规则、实体/伏笔/设定钩子、CLI 审查链、卷级自审、YAML 工具化 |

### 1.1 一句话 premise

> 把小说生成从“贴提纲出稿”升级成“按规则写作、按工具审查、按卷级闭环交付”的故事工程系统。

### 1.2 hook line

> AI 不再只是会写章节，而是知道怎么写、怎么自查、什么时候算一卷完成。

### 1.3 核心卖点

1. 写前、写中、写后规则完整接入，不再只靠 prompt 侥幸出稿。
2. 结构化故事真相层真正服务写作与审查，而不是只做 YAML 存档。
3. 首卷引入、卷级闭环、爽点兑现、AI 风格控制都进入同一条 CLI 闭环。

---

## 2. 背景与问题

当前《命灯照骨》23 章人工审查已明确暴露以下高优先级问题：

- 第一卷尚未形成完整小故事闭环，高潮不等于卷完成。
- 开篇世界观、核心概念、制度压迫逻辑解释不足。
- 首卷疑点、势力、人物、物件堆叠过密，回收不足。
- 整卷长期压抑，爽点与商业承诺不清。
- AI 风格问题已从“个别句子”升级为“高频风格习惯”。
- 现有工作流仍要求 AI 直接读改 YAML，产品闭环不成立。

这些问题说明：

- 单章检测不能替代卷级判断。
- 细纲不能替代成熟叙事结构。
- 故事协议不能只存在于文件里，必须进入写作和审查链。

## 3. 目标用户与场景

### 3.1 目标用户

- 主要用户:
  - 使用 Story Canvas 写长篇网文/长篇小说的作者
  - 负责推进正文的 AI 代理
- 次级用户:
  - 人工审查者
  - 负责维护协议和工作流的产品/开发者

### 3.2 核心场景

1. 作者准备开一本新书，需要明确目标读者、核心体验、首卷职责和写作规则。
2. AI 开始写一章前，需要知道上章结果、本章承接点、本章交付点和禁忌。
3. AI 写完一章或一卷后，需要通过 CLI 得到可执行的修文建议，而不是抽象分数。
4. 作者或审查者要快速核对实体、伏笔、设定、角色、势力、卷级结构是否成立。
5. 长篇项目推进过程中，不希望 AI 直接修改协议 YAML 来维持工作流。

---

## 4. 产品目标

### 4.1 主目标

1. 建立第一版 repo-native 写作规则包。
2. 建立首卷引入与卷级闭环的审查门槛。
3. 建立实体/伏笔/设定钩子进入写作与审查链的机制。
4. 建立 YAML 相关操作的 CLI 能力替代路径。
5. 建立更强的 AI 风格检测与卷级自审输出。

### 4.2 非目标

- 不追求“一键生成即合格”的黑盒写作。
- 不在本轮引入必须联网的核心检测能力。
- 不在本轮重做全部协议格式。
- 不在本轮把 Web UI 作为主交付面。

## 5. 需求模块拆分

### 5.0 总表

| 模块 | 核心用户问题 | 目标能力 | 非目标 |
|------|--------------|----------|--------|
| A. 立项与卷职责定义 | 不知道这个项目、这卷、这章到底要交付什么 | 提供项目级 PRD、卷目标、章承接点、章交付点、压制源与预期爆发点的标准化表达 | 不把自由文本 PRD 变成新的真相源 |
| B. Repo-native 创作操作 | 新章创建、结构同步、状态查看仍依赖直接改 YAML | 提供原子化章节创建、边界回填/校验、状态浏览、卷级导出 | 不把 AI 直接改协议文件当正式工作流 |
| C. 故事真相层钩子 | 实体、伏笔、世界规则虽建档，但没有稳定进入写作与审查 | 提供 mention 扫描、包裹建议、补档提示、伏笔提醒、世界规则写前注入与写后核对 | 不要求所有名词都强制包裹 |
| D. 章节闭环与卷级 Gate | 单章通过不等于整卷可交付 | 明确章节闭环与卷级闭环，支持 AI 卷级自审 -> 修正 -> 复检 -> 人工审查 | 不做多人协作审批系统 |
| E. 审查评分与修复闭环 | review 对张力、闭环、爽点、自审归因不够敏感 | 建立稳定评分维度、卷级审查包、缺陷分类输出和修复建议 | 不承诺高分等于成书质量保证 |
| F. 风格与可读性控制 | AI 味、抽象腔、POV 越界、元信息泄漏持续破坏读感 | 提供可定位、可解释、可修的风格检测与可读性检查 | 不追求抽象“AI 率” |
| G. 导出与人工审查交付物 | 缺少按卷导出、卷级审查包和更可读的实体总览 | 提供按卷命名导出、卷级结构稿、世界/实体百科与变化轨迹 | 不让导出产物反向成为新的真相源 |

### 模块 A: 写作规则包

核心用户问题：

- AI 不知道这一章到底该怎么写、哪些地方不能写偏。

目标能力：

- 明确“要怎么写 / 不要怎么写”
- 规则按类别管理，而不是散落在口头约束里
- 规则能被写前上下文、写中约束、写后审查共同消费

最小范围：

- 开篇规则
- 首卷规则
- POV 规则
- 风格规则
- 角色规则
- 反派规则
- 卷级闭环规则

非目标：

- 不把规则写成纯文学理论长文
- 不要求第一版就覆盖所有题材

### 模块 B: 首卷结构与卷级闭环

核心用户问题：

- 有高潮但卷没收住，首卷不知道该完成什么。

目标能力：

- 明确“卷 = 一个小故事单元”
- 首卷有引入职责、复杂度预算、解释预算、爽点兑现节点
- 卷级闭环成为 gate，而不是靠人工体感判断

最小范围：

- 核心概念最晚解释时点
- 首卷疑点预算 / 解释预算
- 卷主问题 / 阶段性高潮 / 卷尾交付
- 三幕式 / 五幕式等价结构槽位

非目标：

- 不在第一版自动生成完整结构分析报告

### 模块 C: 实体与设定钩子

核心用户问题：

- 实体、角色、势力、物品、设定虽然建档了，但 AI 写作和审查时并没真正用上。

目标能力：

- 已知实体自动给出包裹建议
- 未建档实体给出补档建议
- 当前章相关设定最小注入
- 势力层级、地域扩张、战力阶段进入审查链

最小范围：

- mention 扫描
- 去重反馈
- 引号称呼例外
- 当前章实体摘要
- 当前卷世界尺度摘要

非目标：

- 不在本轮做完整知识图谱

### 模块 D: 伏笔与钩子闭环

核心用户问题：

- 伏笔只是在记账，没有提醒、没有回收压力、没有复杂度控制。

目标能力：

- 区分短/中/长期伏笔
- 支持 due window 提醒
- 未回收进入 warning 或 gate
- 控制首卷疑点与钩子密度

非目标：

- 不追求自动替作者设计全部伏笔

### 模块 E: YAML repo-native 工具化

核心用户问题：

- 新章推进、结构同步、边界回填、状态查看仍依赖 AI 直接改协议文件。

目标能力：

- 让 YAML 操作尽量经由 CLI 完成
- 把“AI 直接读改 YAML”降为例外，而不是主路径

最小范围：

- 新章创建
- 章节结构同步
- scenePlans 边界回填
- 项目/章节/审查状态浏览
- 卷级导出 / 审查包导出

非目标：

- 不在本轮彻底取消所有人工文本编辑

### 模块 F: AI 风格检测增强

核心用户问题：

- 当前 AI 风格问题不是少数禁忌词，而是持续性风格习惯。

目标能力：

- 识别高频支架句风格习惯
- 识别抽象递进、故作深沉、方案文档腔
- 识别元信息泄漏和 POV 越界
- 输出可修文的 evidence 和 suggestion

非目标：

- 不把“黑盒 AI 概率分”当核心结果

### 模块 G: 评分、自审与修复循环

核心用户问题：

- AI 审查仍停留在“跑命令看结果”，不能稳定产出一轮真正的自修闭环。

目标能力：

- 卷级评分维度稳定化
- AI 自审报告格式固定化
- 修文后复检成为标准路径

最小范围：

- 评分维度表
- 自审报告模板
- 缺陷归因模板
- 修复循环路径

非目标：

- 不试图替代人工终审

## 6. 成功标准

### 6.1 功能验收

- AI 不再频繁直接改 YAML 才能推进主线工作流。
- 第一卷能被明确判断为“未闭环”或“已闭环”，而不是只看高潮强度。
- 实体、伏笔、设定、势力等真相层能被 CLI 审查链实际消费。
- 风格检测能稳定命中至少一批人工已确认的高风险样例。

### 6.2 质量验收

- 开篇较早完成核心概念和世界底板 onboarding。
- 首卷复杂度可控，不再持续堆谜不收。
- 卷级至少有一次明确体验兑现，而不只持续压抑。
- AI 风格问题从“通篇统一腔调”下降到“局部可修”。

## 7. 约束

- `base install` 继续保持 stdlib-only。
- 第三方增强只能作为 optional dependency。
- `services/` 保持纯逻辑，`providers/` 负责 optional dependency wrapper。
- 当前阶段优先让 CLI 主闭环成立，UI 只做可视化壳。

## 9. 当前建议

先按以下顺序进入实现：

1. 写作规则包第一版
2. YAML repo-native CLI 能力
3. 实体 / 伏笔 / 设定钩子
4. 卷级闭环 gate 与评分自审
5. AI 风格检测增强

理由：

- 先明确规则，AI 才知道怎么写
- 先补工具，AI 才不用反复直接改协议文件
- 先补卷级 gate，人工审查才不会持续在未完成态介入
