## 1. 背景

当前仓库已经在做两类事情：

1. 把 Story Canvas 收敛成可执行的故事工程工具。
2. 把 skill / adapter 从“纯命令路由”逐步升级为“写法指导 + CLI 路由”。

但当前仍有一个明显缺口：

- skill 和工具规则的大部分写法知识，仍然更多依赖 agent 临场智能与少量经验总结；
- 仓库里还没有一套系统化的、可复查的“从已完成商业作品反推写法”的知识工程；
- `noval-data/` 虽然已有真实书目，但目前还停留在题材级分类，尚未进入“逐书拆解 -> 横向归纳 -> 反哺 skill / tool”阶段。

因此需要立一个更大的工程，用典型商业成书做拆解样本，沉淀成：

- 立项 PRD 写法
- 大纲 / 细纲写法
- 伏笔与回收写法
- 角色塑造与关系推进写法
- 势力 / 世界规则 / 实体维护写法
- 读者吸引机制与商业连载节奏
- 可转写为 prompt / checklist / rule pack 的结构化结论

## 2. 目标

1. 建立一套 repo-native 的“商业成书拆解工作台”。
2. 选取一批典型书籍，按 Story Canvas 自己的创作流程反向拆出：
   - `PRD`
   - `大纲`
   - `细纲 / scene / beat`
   - `伏笔`
   - `角色`
   - `势力 / 世界规则 / 关键实体`
   - `读者钩子`
   - `优点 / 缺点 / 失手点`
3. 在逐书拆解之外，再建立按角度横向归纳的总结层。
4. 最终产出一套可直接反哺 skill / prompt / review 规则的写法方法库。

## 3. 非目标

- 不把原始商业小说文本纳入版本控制。
- 不做“全量逐章精读所有数据集”的无限工程。
- 不把拆解结论包装成唯一真理；允许保留 `TBD`、置信度和争议项。
- 不在本轮直接把全部总结接入 CLI 规则引擎。
- 不把这项工程做成“只生成一份长文报告”，而必须能持续增量维护。

## 4. 目标用户

### 4.1 直接用户

- 维护 `story-harness-writing` / `story-canvas` skill 的开发者
- 负责整理写作规则与提示词的作者
- 负责做真实写作验证和人工审查的人

### 4.2 间接受益者

- 依赖 Skill 写小说的 agent
- 用 Story Canvas 管理长篇项目的作者
- 需要把人工经验回灌到工具的产品 / 开发者

## 5. 核心问题

当前想解决的问题不是“怎样评价一本书”，而是下面这些更工程化的问题：

1. 一本完成度高的商业作品，究竟在 `PRD -> 大纲 -> 正文 -> 审查` 各阶段做对了什么？
2. 这些“做对的地方”里，哪些可以转化为可复用的方法，而不是只适用于某一本书？
3. 哪些失败点是故事本身的问题，哪些是 PRD、细纲、角色控制或伏笔管理不到位？
4. 哪些规律值得进入：
   - skill 写法指导
   - prompt 模板
   - review 维度
   - style / consistency / doctor 的检查项候选

## 6. 范围

### 6.1 本工程分析对象

以 `noval-data/` 中的真实书目为样本池，优先覆盖：

- 玄幻
- 西幻
- 日轻 / 二次元
- 女频关系 / 言情向补样
- 都市超凡
- 历史 / 经营 / 权谋

### 6.2 本工程分析维度

每本典型书至少从以下几个角度拆：

1. `PRD 反推`
   - 面向人群
   - 写作目的
   - 核心承诺
   - 商业卖点
   - 连载模型
2. `大纲与卷结构`
   - 卷级目标
   - 黄金三章
   - 第一卷交付
   - 后续展开方式
3. `细纲与 scene / beat`
   - 单章承接
   - 单章交付
   - 场景切换
   - 章尾钩子
4. `角色塑造`
   - 主角驱动力
   - 配角功能
   - 关系推进
   - 角色辨识度
5. `势力 / 世界规则 / 实体`
   - 势力层级
   - 关键设定 onboarding
   - 组织 / 阵营 / 规则约束
6. `伏笔与线程`
   - 短中长期伏笔
   - 回收节奏
   - 悬念密度控制
7. `写作手法与读者吸引机制`
   - 钩子
   - 反压制 / 爽点
   - 信息投放
   - 对话 / 场景 / 节奏
8. `优点 / 问题 / 可复用结论`
   - 哪些地方特别有效
   - 哪些地方明显失手
   - 哪些经验可以转成 prompt / rule

### 6.3 本工程产物层次

本工程必须同时产出三层结果：

1. `逐书层`
   每本书的详细拆解资料。
2. `横向角度层`
   例如“所有样本的钩子写法总结”“所有样本的主角塑造对比”。
3. `方法沉淀层`
   直接面向 skill / prompt / tool 的写法方法库与规则候选。

## 7. 成功标准

### 7.1 最小成功标准

- 能固定 4-6 本典型书作为第一批样本。
- 每本样本都能按统一模板完成拆解。
- 能从拆解中明确提炼出一批“可写进 skill 的方法”，而不是停留在读后感。
- 能区分：
  - 作品做得好的地方
  - 作品没做好的地方
  - 这些结论中哪些适合工具化，哪些仍应保持人工判断

### 7.2 进阶成功标准

- 至少形成一份跨书横向写作方法总表。
- 至少形成一份 `prompt / skill` 候选规则清单。
- 后续每分析新书时，不需要重想目录和模板，只需要按已有结构增量补充。

## 8. 约束

- 原始 `.txt` 保持本地数据集身份，不纳入版本控制。
- 跟踪产物只保存衍生分析和结构化结论，不大量复制正文。
- 所有判断尽量区分：
  - 事实观察
  - 解释判断
  - 方法建议
- 不把单本书的局部技巧误写成通用定律。
- 结论必须尽量与 Story Canvas 现有流程对齐，而不是重新定义一套并行创作方法论。

## 9. 第一批样本选择原则

第一批不追求面面俱到，而追求“风格差异够大、商业机制够清楚、能快速产出高价值方法”。

建议优先：

1. 玄幻男频爽文
2. 都市超凡
3. 西幻 / 魔幻轻小说
4. 日轻 / 校园恋爱喜剧
5. 女主修仙 / 关系驱动作品

样本选择标准：

- 类型有代表性
- 开篇钩子明确
- 卷级结构可拆
- 优缺点都足够明显
- 能服务后续 skill 收敛

## 10. 阶段拆分

### Phase 0: 规则与模板先行

- 固定目录结构
- 固定拆解模板
- 固定证据记录方式
- 固定横向归纳口径

### Phase 1: 第一批典型书逐书拆解

- 先做 4-6 本
- 每本都完成逐书分析包
- 先求完整，不追求所有维度无限深

### Phase 2: 横向方法归纳

- 按 `PRD / 钩子 / 主角 / 伏笔 / 势力 / 节奏 / 关系 / 爽点` 做跨书总结

### Phase 3: 反哺 Skill / Tool

- 把高价值方法整理成：
  - skill 写法指导
  - prompt 片段
  - review / style / doctor 候选规则

## 11. 风险

- 如果没有统一模板，后续很快会变成杂乱读书笔记。
- 如果只做逐书分析，不做横向归纳，就无法真正反哺 skill。
- 如果过早直接写规则，不先保留证据链，后续难以复审。
- 如果不区分“适合工具化”和“保持人工判断”，系统会变得又重又僵。

## 12. 当前结论

这项工程的本质，不是“做一份商业小说赏析”，而是：

- 以商业成书为训练材料
- 以 Story Canvas 自己的流程为拆解框架
- 以 skill / prompt / tool 反哺为最终目标

也就是说，它应被视为：

`真实作品分析工程` -> `写法知识工程` -> `工具与 skill 能力底盘`
