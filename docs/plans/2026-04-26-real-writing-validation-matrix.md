# v1.0 真实写作验证矩阵

> 日期: 2026-04-26
> 状态: 草稿
> 关联当前入口: `docs/roadmap.md`
> 关联计划:
> - `docs/plans/2026-04-26-writing-gap-remediation-plan.md`
> - `docs/plans/2026-04-26-writing-review-gap-log.md`

## 1. 背景

当前仓库已经有短篇、轻小说、西幻、都市志怪等 demo 与 smoke tests，但这些更多是在验证：

- 命令闭环能不能跑通
- review / workflow / export 输出结构是否稳定
- 少量题材信号能否被系统消费

这还不等于 `v1.0` 已经通过真实写作验证。

在发布前，必须引入固定的、多风格、多卷级测试矩阵，避免“只在单一题材样例上表现不错，就误判产品已成熟”。

## 2. 目标

1. 在 `v1.0` 发布前建立稳定的真实写作测试矩阵。
2. 固定至少四类典型风格：
   - 玄幻
   - 西幻
   - 日轻
   - 言情
3. 每类风格至少完成 `2` 卷级别测试；更优目标是其中至少 `2` 类达到 `3` 卷。
4. 把人工审查发现的问题稳定回灌到：
   - `style`
   - `consistency`
   - `review`
   - `doctor`
   - `workflow`
5. 验证系统是否已经形成“写前约束 -> 写中遵循 -> 写后检测、自审、修正”的真实质量闭环。

## 3. 非目标

- 不要求四类风格都先写成完整长篇成书。
- 不把“真实写作验证”误写成纯自动化测试。
- 不在这一轮追求题材覆盖无限扩张。
- 不在这一轮把每个问题都立刻产品化；先记录、归类、收敛高价值缺口。

## 4. 测试矩阵

### 4.1 固定风格

| 风格 | 最低卷数 | 更优卷数 | 当前用途 |
|------|------|------|------|
| 玄幻 | 2 卷 | 3 卷 | 验证世界规则、境界逻辑、成长线、伏笔兑现；优先参考 `demo-guixu` |
| 西幻 | 2 卷 | 3 卷 | 验证设定清晰度、群像关系、异世界背景信息投放与长线冲突 |
| 日轻 | 2 卷 | 3 卷 | 验证角色魅力、轻快节奏、场景感、对话驱动和读者追更欲望 |
| 言情 | 2 卷 | 3 卷 | 验证情感张力、关系递进、情绪契约兑现、人物塑造稳定性 |

### 4.2 卷级目标

- 第一卷:
  - 必须重点检查“黄金三章”
  - 要回答：读者会不会愿意继续读
  - 要检查：
    - 钩子是否成立
    - 主角处境是否明确
    - 核心承诺是否尽早出现
    - 节奏是否拖沓
    - 风格是否快速成型
- 第二卷:
  - 必须重点检查“展开是否站得住”
  - 要检查：
    - 世界与背景逻辑是否自洽
    - 第一卷承诺是否延续而非失踪
    - 人物关系是否进入推进而非原地踏步
    - 伏笔、悬念、线程是否有回收节奏
    - 是否出现设定补丁式硬修
- 第三卷（如有）:
  - 重点检查长线稳定性
  - 要检查：
    - 状态追踪是否持续可用
    - 人物弧线是否出现断裂
    - 子冲突与主线是否开始互相支撑
    - review / workflow / context 是否仍能服务复杂项目

## 5. 核心审查维度

### 5.1 第一层: 是否想继续读

- 首三章是否形成明确阅读欲望
- 章尾是否有继续阅读驱动力
- 是否存在明显“方案文档腔”或 AI 套话
- 是否过早堆砌设定而削弱故事吸引力

### 5.2 第二层: 故事与背景逻辑

- 世界规则是否前后一致
- 角色能力、身份、任务风险是否匹配
- 设定揭露是否自然，是否存在“无来源姓名/身份直接抛出”
- 时间线、因果链、伏笔回收是否能持续追踪
- 章与章之间是否自然承接，而不是只在每章内部自洽

### 5.3 第三层: 人物塑造

- 主角目标、压力与选择是否清楚
- 角色关系是否有动态变化，而非标签化摆设
- 情绪反应是否稳定，不随剧情硬拐
- 关键角色是否在多章后仍保有辨识度

### 5.4 第四层: 工具是否真的帮上忙

- `doctor` 是否能提前指出高价值缺口
- `style` 是否能抓到风格跑偏或腔调问题
- `consistency` 是否能抓到真正危险的设定漂移
- `review` 是否能输出对作者有用的优先动作
- `workflow` 是否能帮助推进，而不是增加负担
- CLI 检测后，是否真的能支撑“自审 -> 修正 -> 复检”，而不是只产出一份报告

### 5.5 第五层: 章间衔接与 AI 味

- 当前章开头是否接住上一章的结果、情绪或行动后果
- 当前章是否只是把大纲/细纲事项机械写成正文
- 是否存在明显程式化过渡、方案文档腔、说明替代表演等重 AI 味问题
- 这些问题是否已被 CLI 提前发现，还是仍然只能靠人工读感发现
- 若正文偏离细纲，偏离是否让人物和故事更合理，还是只是生成跑偏

## 6. 每轮测试的最小产出

每个测试项目至少要保留以下产物：

1. 项目根目录样例
2. 对应章节正文
3. `review chapter` / `review scene` 结果
4. `doctor` / `consistency` / `style` 输出快照
5. 人工审查记录
6. 问题回灌结论：
   - 属于误报
   - 属于漏报
   - 属于协议缺口
   - 属于 workflow/命令可用性问题

## 7. 人工审查记录格式

建议每轮固定记录以下内容：

```text
项目：
风格：
卷号：
章节范围：

黄金三章结论：
继续阅读欲望：

故事/背景逻辑问题：
章间衔接问题：
人物塑造问题：
风格/腔调问题：
工具已发现问题：
工具漏掉问题：
细纲偏离情况：
偏离是否合理：

建议进入系统的问题：
建议保持人工判断的问题：
```

## 8. 与现有样例的衔接

### 8.1 当前可直接复用

- `projects/demo-short-story/`
- `projects/demo-light-novel-short/`
- `projects/demo-xuanhuan-short/`
- `projects/demo-urban-occult-long/`

### 8.2 仍需补齐

- 真正用于 `v1.0` 发布前人工写作审查的固定矩阵项目
- 至少一个玄幻复杂长篇视角，优先参考 `demo-guixu`
- 至少一个卷级言情样例
- 至少一个卷级西幻长线样例

## 9. 执行顺序建议

1. 先把当前功能与检测继续收敛到“可稳定跑样例”
2. 再固定四类风格的测试项目骨架
3. 先跑第一卷，重点审“黄金三章”
4. 再推进第二卷，重点审逻辑与人物稳定性
5. 每轮人工审查后，把问题写入 `writing-review-gap-log`
6. 只有当同类问题反复出现时，才提升为系统级规则或协议改动
7. 对复杂长篇样例，优先检查是否形成“检测 -> 自审 -> 修正 -> 复检”的循环，而不是只比较单次输出好坏
8. 若发现“细纲原样执行反而更不合理”，应记录这是角色/衔接/世界规则约束胜过执行路径，还是模型单纯跑偏

## 10. 发布前最低门槛

- 四类典型风格均完成至少 `2` 卷级写作测试
- 每类至少经过一轮人工审查
- `writing-review-gap-log` 中的高频问题有明确归因：
  - 已修
  - 延后到 `v1.1+`
  - 保持人工判断
- 至少一个复杂长篇项目证明系统不是只适合玩具样例

## 11. 增强门槛

- 至少 `2` 类风格达到 `3` 卷
- 至少一个项目完成“第一卷建立兴趣 -> 第二卷展开故事”的连续验证
- 人工审查发现的问题，能稳定回灌成可回归验证的样例或 smoke test

## 12. 风险

- 如果只写少量短样例，会误判长篇能力已经稳定
- 如果没有人工审查，会把“工具能输出东西”误判成“故事真的可读”
- 如果风格矩阵不固定，后续改动会持续漂移，无法判断回归
- 如果问题不做归因，团队会陷入反复盲改
- 如果不单独检查章间衔接，会把“每章局部过关”误判成“整卷阅读体验成立”

## 13. 当前结论

`v1.0` 发布前的“样例矩阵”不应再被理解为：

- 只要一短一长能跑通
- 只要 demo 能导出
- 只要 review 有分数

而应被理解为：

- 多风格
- 多卷级
- 带人工审查
- 能持续把真实问题回灌到系统
