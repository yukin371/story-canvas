# 《命灯照骨》23 章人工审查后的问题与解决方案总整理

> 日期: 2026-04-27
> 状态: 草稿
> 关联当前入口: `docs/roadmap.md`
> 关联文档:
> - `docs/plans/2026-04-26-writing-review-gap-log.md`
> - `docs/plans/2026-04-26-workflow-gap-notes.md`
> - `docs/plans/2026-04-26-writing-quality-loop.md`

## 1. 目标

把这轮 23 章人工审查结论收敛成一份后续执行总整理，明确回答四件事：

1. 以后给 AI 的写作约束应是什么
2. CLI / repo-native 工具应补哪些能力
3. 实体、伏笔、设定、角色、势力等“故事钩子”应如何进入审查链
4. 评分、自审、循环修复应如何升级成可执行闭环

## 2. 总结论

当前项目已经证明：

- AI 能持续产出章节
- 也能在局部制造张力、氛围和高潮

但同样已经明确暴露：

- 仅靠大纲和细纲，不等于能写成完整小说
- 仅靠单章检测，不等于能保证卷级闭环
- 仅靠少量 style heuristics，不足以压住持续性的 AI 风格习惯
- 仅靠 agent 手工读写协议文件，不是可交付的产品工作流

后续应同时收敛三条主线：

1. **写作约束主线**
   - 让 AI 在写前知道规则、写中守住边界、写后进入自审与修正
2. **工具主线**
   - 尽量把 YAML 驱动能力收回到 repo-native CLI，而不是依赖 AI 直接读改协议文件
3. **质量闭环主线**
   - 把“风格、设定、伏笔、实体、结构、商业承诺”都转成可审查、可评分、可回灌的问题链

## 3. 问题分层

### 3.1 写作侧问题

- 首卷引入职责不清
- 世界观和核心概念 onboarding 不足
- 制度压迫与角色动机链不自洽
- 反派/压迫方只有功能，没有足够逻辑和人格
- 疑点、势力、人物、物品堆叠过密
- 高潮有了，但第一卷仍未形成小故事闭环
- 整卷长期压抑，爽点和爆发兑现不足

### 3.2 风格侧问题

- 高频支架句风格习惯严重
- 抽象递进、故作深沉腔明显
- 元信息泄漏进正文
- POV 可知性越界
- 方案文档腔、提纲执行稿感

### 3.3 工具侧问题

- 新章创建、结构同步、段落边界回填仍依赖直接改 YAML
- 项目/章节/审查状态浏览不够好
- 实体包裹和实体建档缺少闭环
- 伏笔账本只有台账，没有到窗提醒与回收 gate
- 缺少卷级审查包、卷级导出、卷级结构稿
- 缺少“卷已完成”的真正 gate

### 3.4 审查闭环问题

- 单章 review 不能替代卷级判断
- 当前评分对张力、爽点、闭环、复杂度预算仍不够敏感
- AI 自审仍容易停留在“跑命令 + 读结果”，缺少系统修文策略

## 4. 给 AI 的写作约束

这一层回答“要怎么写，不要怎么写”。

### 4.1 要怎么写

- 先确认当前卷职责，再写当前章。
- 先确认本卷主问题、本章承接点、本章交付点，再动笔。
- 让故事随着人物成长，但不能偏离当前卷的大局交付。
- 细纲是执行路径，不是最高真相源；若人物自然反应与细纲冲突，应优先守住：
  - 已发生后果
  - 世界规则
  - 人物当前状态
  - 关系逻辑
  - 本章必须交付的推进
- 把人物反应写具体：
  - 优先动作
  - 优先感官
  - 优先明确判断
  - 少做抽象概念递进
- 在首卷前段循序渐进展开：
  - 先抓少数核心人物
  - 先抓少数核心势力
  - 先抓少数核心疑点
  - 给阶段性解释和阶段性回报
- 让敌对方像真人或真实势力：
  - 有目标
  - 有利益
  - 有边界
  - 有手段风格
  - 不只是给主角添麻烦
- 让每一卷像一个小故事：
  - 有卷主问题
  - 有阶段性高潮
  - 有一次明确交付
  - 有卷尾余波

### 4.2 不要怎么写

- 不要把细纲事项直接翻成正文。
- 不要为了制造神秘感而无止境加谜。
- 不要把首卷写成高密度悬疑过载铺陈。
- 不要让整卷一直“压”，却迟迟不给“扬”。
- 不要让人物知道他不可能知道的信息。
- 不要让章节、卷、提纲、工作流语言漏进正文。
- 不要高频使用统一的 AI 支架句习惯，例如：
  - `不是……是……`
  - `不是……不是……是……`
  - `不是……更像……`
  - 高频 `像……`
  - `不重/不轻/不偏/不倚`
  - `真正……的，从来都是……`
  - `还有什么？`
- 不要把人物即时反应写成抽象、伪深刻、故作深沉的概念句。
- 不要默认所有实体都靠 AI 自己临时记忆维持，应依赖结构化信息。

### 4.3 应形成的写作规则包

后续应把上述要求收敛成 repo-native 写作规则包，至少分为：

- 开篇规则
- 首卷规则
- POV 规则
- 风格规则
- 角色规则
- 反派规则
- 实体包裹规则
- 伏笔与回收规则
- 卷级闭环规则

## 5. CLI / Tooling 改进方向

这一层回答“哪些能力不该继续靠 AI 直接改文件完成”。

### 5.1 总原则

- `md` 类自由文本整理，可接受 agent 自带编辑工具辅助处理。
- `yaml` / 协议状态文件，应尽量由 repo-native CLI 能力负责创建、更新、校验、导出。
- 不应把“AI 能直接读改 YAML”当成正式工作流前提。

### 5.2 需要优先工具化的 YAML 相关能力

#### P0

- 新章创建命令
  - 原子完成：
    - 新建正文
    - 创建 outline direction / beats / scenePlans
    - 推进 `activeChapterId`
- 场景边界回填 / 重算命令
- 项目状态 / 章节状态 / 审查状态浏览命令
- 卷级导出命令
  - 支持 `第一卷.md` / `第一卷.txt`
- 卷级审查包导出

#### P1

- 卷级大纲/细纲收敛稿导出
- 结构模板驱动的卷创建或卷检查
- 伏笔到窗提醒与未回收告警
- 实体对齐 / 实体补档 / 实体去重提示命令

#### P2

- 顺序化工作流编排入口
- 批量运行写后审查链
- 自动汇总修正建议与复检结果

## 6. 实体、伏笔、设定、角色、势力等钩子功能

这一层回答“故事真相层如何真正帮助写作和审查”。

### 6.1 实体钩子

应形成一条完整闭环：

1. 扫描当前章 mention
2. 已知实体给出包裹建议
3. 未建档实体给出补档建议
4. 同一实体在同章只反馈一次
5. 向 AI 注入当前章相关实体的最小设定摘要

注意例外：

- 引号内称呼默认降级处理
- 别名、职位称呼、情绪化叫法不应机械强制包裹

### 6.2 伏笔钩子

应把伏笔从“存在台账”升级为“可写、可审、可追责”：

- 区分短期 / 中期 / 长期伏笔
- 明确 planted / due / resolved 状态
- 在章节与卷级审查中提醒到窗项
- 到窗未回收时给 warning 或 gate 阻塞
- 同时控制首卷疑点预算，避免无止境堆叠

### 6.3 设定钩子

应让 worldbook 真正进入写作闭环，而不是只做静态存档：

- 核心概念最低定义
- 制度运转条件
- 势力层级关系
- 地域扩张路径
- 战力阶段与冲突边界
- 主角成长后舞台扩张预期

### 6.4 角色与反派钩子

角色侧至少应显式维护：

- 当前状态
- 核心动机
- 当前压力
- 关系变化
- 阶段性成长

敌对方至少应显式维护：

- 目标
- 利益
- 约束
- 手段风格
- 人格辨识度

## 7. AI 检测改进方向

这一层回答“如何更有效地控制 AI 风格问题”。

### 7.1 检测目标

不是追求一个抽象的“AI 率”，而是抓出可修的问题：

- 高频支架句风格习惯
- 抽象递进和故作深沉句
- 方案文档腔
- 提纲执行稿感
- 元信息泄漏
- POV 越界
- 角色口吻混同

### 7.2 技术路线

- `base` 保持 builtin heuristics
- 外部增强仅作为 optional dependency
- 重点增强：
  - 句法骨架重复
  - 邻近段落重复密度
  - 支架句组合式复用
  - 角色口吻归属统计

### 7.3 输出要求

检测输出不能只说“像 AI”，必须尽量回答：

- 哪一类问题
- 在哪里
- 为什么算问题
- 建议如何改

## 8. 评分、自审、循环修复标准

这一层回答“怎么让 AI 不只是跑命令，而是真正完成一轮自审”。

### 8.1 评分维度

后续卷级评分至少应稳定覆盖：

- 世界观 onboarding
- 核心概念清晰度
- 首卷复杂度控制
- 伏笔与回收节奏
- 章间承接
- POV 一致性
- 角色塑造
- 反派逻辑
- 势力/设定遵循
- 张力与冲突升级
- 爽点引擎与兑现
- 卷级闭环完成度
- AI 风格风险
- 手机端阅读负担

### 8.2 自审输出格式

AI 完成一卷后，不应直接交稿，应至少输出：

1. 本卷主问题是否收束
2. 本卷主爽点是否兑现
3. 当前最大 5 个缺陷
4. 哪些是生成漏做
5. 哪些是自审漏看
6. 哪些是工具漏报
7. 准备如何修

### 8.3 修复循环

标准路径应为：

1. 生成
2. `style`
3. `consistency`
4. `review chapter`
5. `review volume` / 等价卷级审查
6. AI 输出自审报告
7. AI 修文
8. 复检
9. 人工审查

## 9. 结构模板与作家写法融入方向

“融入作家的写作手法”不应理解成模仿名句，而应理解成显式消费成熟叙事方法。

后续应至少支持：

- 三幕式
- 五幕式
- 引入卷结构模板
- 黄金开篇检查表
- 小故事单元闭环模板

并把它们转成工具可消费的信息：

- 当前卷在哪一幕
- 这一幕应完成什么职责
- 哪些信息必须解释
- 哪些谜题可以延后
- 哪些情绪必须兑现

## 10. 优先级建议

### P0

- 写作规则包第一版
- 新章创建 / 状态浏览 / 边界回填 / 卷导出
- 首卷复杂度预算
- 核心概念 onboarding 检查
- 高频支架句风格习惯检测
- 卷级闭环 gate

### P1

- 实体包裹建议 + 实体摘要注入
- 伏笔到窗提醒与回收 gate
- 爽点引擎 / 核心体验 / 目标读者结构槽位
- 反派逻辑与势力层级审查
- 卷级结构模板与卷级结构审查

### P2

- 更强的 optional AI detection enhancement
- 章节 / 卷级顺序化编排入口
- 面向人工审查的更完整审查包

## 11. 验收标准

后续可视为明显收敛的标志至少包括：

- AI 不再频繁直接改 YAML 才能推进主工作流
- 第一卷能独立形成一个小故事闭环
- 开篇阶段读者能较早理解核心概念与世界底板
- 首卷既有压抑，也有明确兑现
- AI 风格问题从“通篇都是”下降到“局部可修”
- 实体、伏笔、设定不再只存在于文件里，而是真正进入写作和审查链

## 12. 当前执行建议

按顺序推进：

1. 先把这轮审查结论转成第一版写作规则包
2. 再补 YAML 相关的 repo-native CLI 能力
3. 再补实体 / 伏笔 / 设定钩子
4. 再补卷级评分、自审、闭环 gate
5. 最后继续增强 AI 风格检测

原因：

- 先有规则，AI 才知道怎么写
- 先有工具，AI 才不用反复手工改协议文件
- 先有卷级 gate，人工审查才不会总是在未完成状态介入
