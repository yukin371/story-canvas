# 2026-04-26 Workflow Gap Notes

## 背景

在《命灯照骨》长篇样例的真实写作闭环中，新增章节与回填 `scenePlans` 段落边界时，当前流程仍依赖 agent 直接编辑项目内的 `project.yaml`、`outline.yaml`、`detailed_outlines.yaml`。

## 记录事项

- 这说明当前 Story Harness CLI 对“章节创建 / 结构同步 / 段落边界回填”的产品化支撑还不够完整。
- 也说明仓库不能假设调用方拥有 Serena 之类额外 MCP 读写能力；需要 repo-native 的可审查工具链来完成：
  - 项目状态读取
  - 章节结构创建
  - `scenePlans` 边界同步
  - 人工审查辅助与差异核对
- 该问题已记录，但本轮先不解决；等卷级内容闭环和真实测试样例继续推进后，再统一回到工具能力设计。

## 2026-04-26 新增实测问题

### 1. 新章创建仍依赖直接改协议文件

- 现象:
  - 在《命灯照骨》推进到 `chapter-021` 时，仍需要 agent 直接修改：
    - `project.yaml`
    - `outline.yaml`
    - `detailed_outlines.yaml`
- 说明:
  - 当前缺少一个 repo-native 的章节创建/推进入口，无法原子化完成：
    - 新建章节正文
    - 创建章节 direction / beats / scenePlans
    - 推进 `activeChapterId`
- 结论:
  - 这不是正常工作流，而是明确的产品能力缺口。

### 2. `scenePlans` 段落边界没有可靠回填工具

- 现象:
  - 正文一旦修改，`startParagraph / endParagraph` 就要手工重算并同步到结构文件。
- 说明:
  - 当前缺少可靠的 boundary sync 能力，导致样例回归和人工审查都更脆弱。
- 结论:
  - 需要 repo-native 的段落边界回填/重算命令，而不是依赖 agent 手工核对。

### 3. 状态查看仍过度依赖直接翻协议文件

- 现象:
  - 写作过程中需要反复直接查看：
    - `project.yaml`
    - `outline.yaml`
    - `detailed_outlines.yaml`
    - `context-lens.yaml`
    - `story-reviews.yaml`
- 说明:
  - 当前 CLI 缺少足够好的项目状态、章节状态、审查状态浏览命令。
- 结论:
  - 作者不该靠直接翻原始协议文件理解当前进度和上下文。

### 4. 中文短句下的 `unintroducedNameReveal` 误报仍明显

- 现象:
  - 在 `chapter-021` 中，检测器把“石面冷硬”“冷得不对”“第三索”“开什么”误判成姓名揭露。
- 说明:
  - 规则方向是对的，但中文短段落/短句场景下的命名边界识别还不稳。
- 结论:
  - 需要继续降低中文 prose 上的误报率，否则会污染 review 信号。

### 5. `review chapter` 对动作型章节的张力识别失真

- 现象:
  - `chapter-021` 明明存在：
    - 锁梁回紧的时限
    - 守梁役阻拦
    - 一号补完风险
    - 追兵压近
    - 真页代价余波
  - 但 `review chapter` 仍把该章判断为“冲突张力偏平”。
- 说明:
  - 当前评分器对中文短段、动作驱动型章节的冲突信号提取不够稳定。
- 结论:
  - 评分链还不能机械相信，需要后续补强对动作张力与时限压力的识别。

### 6. `chapter analyze` 对关键临时对象覆盖不足

- 现象:
  - `chapter-021` 中，一号是核心行动对象，但分析结果仍主要稳定识别到沈照、岳怀川。
- 说明:
  - 临时对象、旧流程对象、未完全实体化的关键角色，还不能稳定进入分析链。
- 结论:
  - 这会限制后续 `review / projection / consistency` 的感知质量。

### 7. 建议链在真实写作中经常空转

- 现象:
  - 本章 `chapter suggest -> review apply -> projection apply` 基本无新增建议或应用结果。
- 说明:
  - 如果这条链长期空转，就说明当前建议生成要么太弱，要么缺少足够清晰的“本次无需建议”的解释。
- 结论:
  - 需要后续明确这条链在真实写作闭环中的产品价值，而不是只保留命令存在。

### 8. `scenePlans` 存在 off-by-one 边界失配时会直接卡死 review gate

- 现象:
  - `chapter-022` 的第三幕 `endParagraph` 被写成 `473`，而当前正文实际只有 `472` 个段落。
  - `review scene --chapter-id chapter-022 --scene-index 3` 因此直接报 `段落范围无效，可用范围为 1..472`。
- 说明:
  - 这不只是“需要手工回填边界”的老问题，还意味着当前流程缺少最基本的边界合法性兜底。
  - 一旦正文在评审前有微调，显式 `scenePlans` 就可能把整个 review gate 卡住。
- 结论:
  - 需要 repo-native 的边界校验与自动收正能力，至少在运行 review 前给出可修复建议，而不是让作者手工翻协议文件排错。

### 9. 缺少卷级审查包导出，人工审查前仍需人工拼材料

- 现象:
  - 在《命灯照骨》到达当前卷收口点后，CLI 已能导出：
    - 单章 `review-packet`
    - 全书 `markdown`
    - 大纲与角色导出
  - 但仍没有一个 repo-native 的“卷级审查包”把以下内容聚合到一起：
    - 卷范围与卷目标
    - 卷内关键章节承接
    - 最新 review / style / consistency 信号
    - 人工审查前的 AI 自审结论
- 说明:
  - 目前只能退回“导出整卷稿 + 人工自己拼状态”的方式进入人工审查。
- 结论:
  - 这已经不是体验优化，而是卷级闭环工具化缺口。

### 10. 多命令顺序编排摩擦仍会打断真实写作闭环

- 现象:
  - 在 `chapter-023` 修文后，连续执行 `chapter analyze`、`chapter suggest` 时，`chapter suggest` 因“项目状态已被其他命令更新，请重新执行当前命令以避免覆盖”而中断。
- 说明:
  - 这类并发保护本身是合理的，但也暴露出：当前真实写作闭环仍要求操作者自己精确掌握命令顺序和状态刷新时机。
  - 一旦命令链较长，使用者就容易被迫停下排查“该先跑什么”。
- 结论:
  - 后续需要更强的 repo-native orchestration，例如：
    - 明确的串行执行入口
    - 更清楚的“上一步修改了哪些状态、下一步应重跑什么”提示
    - 面向真实写作闭环的批量/顺序化命令组合

### 11. `characters` 导出当前只消费 `entities.yaml`，导致角色/地点/势力覆盖面严重不足

- 现象:
  - 《命灯照骨》的 `characters` 导出目前只列出：
    - 4 个主角色
    - 2 个地点
    - 1 个推断角色
  - 但项目真实协议里还存在：
    - `worldbook.factions`
    - `worldbook.locations`
    - `worldbook.artifacts`
    - `worldbook.mysteries`
    - 更多正文中已经活跃、但尚未沉淀进 `entities` 的对象
- 现有实现原因:
  - `src/story_harness_cli/commands/export.py::_generate_spec_characters()` 当前只读取 `state["entities"]["entities"]`。
  - 因此它天然不会把 `worldbook`、`threads`、`foreshadowing` 等其他真相层对象纳入导出。
- 为什么这轮没解决:
  - 生成阶段没有自动把所有重要对象都沉到 `entities`。
  - 自审阶段没有“导出覆盖率”检查，导致没有在交付前被主动发现。
  - 工具侧也没有“entities 覆盖不足 / worldbook 未进入导出”的诊断提示。
- 结论:
  - 当前 `characters` 更接近“entities 卡片导出”，还不是“项目角色/设定总览导出”。
  - 后续需要明确：
    - 是拆成 `characters / factions / locations / artifacts` 多类导出
    - 还是改成统一的世界/实体百科导出

### 12. `characters` 导出的“首次出场”字段实际是注册时间，不是章节出场信息

- 现象:
  - 导出文件里显示的是：
    - `首次出场: 2026-04-26T17:20:00+08:00`
  - 这对作者审查几乎没有帮助；真正需要的是：
    - 首次出场章节
    - 关键出场章节
    - 高频角色的状态/关系变化节点
- 现有实现原因:
  - `export.py` 当前直接拿实体的 `registeredAt` 输出为“首次出场”。
  - `entities.yaml` 也没有稳定沉淀“首次登场章节 / 章节索引 / 变化里程碑摘要”的统一字段。
- 为什么这轮没解决:
  - 生成阶段主要关注正文推进，没有同步把登场和变化轨迹结构化。
  - 自审阶段没有把导出可读性当作固定检查项。
  - 工具侧没有基于 `changeLog`、章节分析结果和 mention 分布生成“章节出场轴”。
- 结论:
  - 当前字段命名会误导使用者，后续至少要改成：
    - `首次登记时间`
    - 或真正提供 `首次出场章节`
  - 更优目标是补一份“角色变化轴 / 关系变化轴”，而不是把所有变化硬塞进单卡片正文。

### 13. `characters` 导出缺少实体分离、别名治理和异常名称修正支持

- 现象:
  - 当前导出里既有正式角色，也有：
    - `inferred::周小棘`
    - 仅推断出的地点
  - 一旦正文里出现别名、误识别、临时称呼或拆分不清，导出结果就会显得混乱。
- 现有实现原因:
  - 现有导出基本按 `entities` 现状直出，缺少：
    - alias 合并
    - 疑似重名 / 误识别提示
    - inferred 条目与 canon 条目的分层呈现
- 为什么这轮没解决:
  - 生成时没有严格把新对象及时正规化。
  - 自审时没有专门看导出的实体清单是否可用。
  - 工具侧也没有“导出前实体体检”来提示哪些条目应合并、拆分或补档。
- 结论:
  - 后续需要在导出前增加实体治理层，而不是默认 `entities.yaml` 当前状态就适合给人读。

### 14. `global-outline` 对长篇卷级审查仍然过粗，缺少“卷完成后先粗后细”的收敛产物

- 现象:
  - 当前 `global-outline` 能按卷列出章节，但章摘要仍偏一句话压缩。
  - 对人工卷审来说，这种粒度不足以快速把握：
    - 单卷内部的起承转合
    - 本卷承诺与兑现
    - 卷内章节间的因果推进
- 为什么这轮没解决:
  - 生成阶段主要围绕单章方向、beats 与 scenePlans 推进，没有反向沉淀“卷级摘要层”。
  - 自审阶段缺少“卷完成后输出一次卷级大纲/细纲收敛稿”的固定步骤。
  - 工具侧虽然有 `global-outline` / `detail` 导出，但没有专门服务卷审的中间粒度导出。
- 结论:
  - 后续应考虑把流程明确成：
    - 先完成一个卷级别的大纲/细纲收敛
    - 再继续细化下一单元
  - 工具侧则需要补：
    - 卷级摘要导出
    - 卷级细纲导出
    - 面向人工卷审的中粒度审查稿

### 15. 卷导出缺少显式的“按卷导出 + 卷名命名”能力

- 现象:
  - 当前项目可导出整书 `markdown/txt`，但在人工审查当前卷时，更自然的交付物应该是：
    - `第一卷.md`
    - `第一卷.txt`
  - 目前只能导出整书，或手工给导出文件起名。
- 为什么这轮没解决:
  - 生成阶段默认按项目整体推进，没有显式卷级交付动作。
  - 自审阶段没有“当前卷导出是否独立可审”的固定检查。
  - 工具侧 `export` 缺少：
    - `--volume-id`
    - 按卷过滤章节
    - 使用卷标题命名输出文件
- 结论:
  - 这会增加人工审查摩擦。
  - 后续应补明确的卷导出能力，而不是把整书导出和当前卷审混在一起。

### 16. 卷级/章级细纲缺少显式的“爽点、波折、压抑、爆发”结构槽位

- 现象:
  - 当前 outline / detail 更偏事件推进、方向摘要与场景计划。
  - 但对玄幻这类强题材项目，审查与自查时真正想快速看到的往往是：
    - 本卷主爽点引擎
    - 本章压抑来源
    - 本章关键波折
    - 本章或本卷爆发点
- 为什么这轮没解决:
  - 生成阶段默认依赖自由文本 `direction / beats / scenePlans`，没有把这些节奏要素做成显式字段。
  - 自审阶段也缺少一份“本卷/本章节奏设计表”，导致 AI 很难系统化检查“压抑是否铺够、爆发是否兑现”。
  - 工具侧当前没有专门消费或导出这类节奏结构。
- 结论:
  - 后续应考虑在卷级或章级细纲里增加更清楚的节奏信号层，至少让作者和 AI 都能快速审：
    - 爽点承诺
    - 压抑来源
    - 波折节点
    - 爆发/反压制落点

### 17. 缺少“卷完成后先收一次卷级大纲/细纲”的固定流程

- 现象:
  - 当前流程更像持续按章推进，再在末端导出与审查。
  - 但从人工卷审角度，更自然的路径应是：
    - 先把本卷剧情走向、背景底板、冲突升级、爆发落点整理成一版卷级大纲/细纲收敛稿
    - 再让人和 AI 基于这份收敛稿审正文是否兑现
- 为什么这轮没解决:
  - 当前 workflow 还没有真正的卷级 gate。
  - 导出层也缺少面向卷审的“中粒度结构稿”。
  - 自审阶段因此更容易直接面对正文，而不是先面对结构承诺。
- 结论:
  - 这应作为后续卷级闭环的重要一步，而不是继续只靠章级结构散落支撑卷审。

### 18. 伏笔账本已有协议，但缺少“长短期伏笔 + 到窗提醒 + 未回收审查”闭环

- 现象:
  - 当前项目已经有 `foreshadowing.yaml`、`context dueForeshadows`、`foreshadow plant/resolve/list`。
  - 但从真实使用角度，仍缺少更强的审查闭环：
    - 区分长期伏笔与短期伏笔
    - 在适当章节窗口提醒 AI “该回收了”
    - 当应回收未回收时，给出明确 warning 或 gate 阻塞
- 为什么这轮没解决:
  - 生成阶段虽然能记录伏笔，但并不会自动把“本章应回收哪些伏笔”变成强提示。
  - 自审阶段目前也没有系统追问“这一章/这一卷有哪些到窗伏笔未动”。
  - 工具侧虽然能在 `context` 中暴露 `dueForeshadows`，但缺少更可执行的 review / workflow 消费方式。
- 结论:
  - 当前伏笔协议还偏“台账存在”，未真正进入“可审查、可提醒、可追责”的状态。
  - 后续可考虑：
    - 明确 `short / mid / long` 或等价窗口分类
    - 在 `review chapter / review volume` 中输出“应回收未回收”的审查项
    - 在 workflow gate 或 context refresh 中给出更强提示，而不是只静态展示

### 19. 当前项目初始化流程没有把 `PRD.md` 变成默认存在的立项入口

- 现象:
  - 仓库中已经存在：
    - `docs/templates/PRD.template.md`
    - `projects/demo-urban-occult-long/PRD.md`
    - `projects/demo-guixu/PRD.md`
  - 但当前真实项目 `projects/mingdengzhaogu/` 没有 `PRD.md`。
  - 与此同时，`project.yaml` 虽然已经承载了 `positioning / storyContract / commercialPositioning`，但这仍不足以替代完整的立项文档。
- 现有实现原因:
  - 当前 `init` / 项目流程主要落 `project.yaml` 协议字段，没有把 `PRD.md` 设为默认产物或强制门禁。
  - `doctor` 目前也不会因为缺少 `PRD.md` 而给出显式提示。
- 为什么这轮没解决:
  - 生成阶段直接沿用 `project.yaml + outline` 推进，没有先补正式 PRD。
  - 自审阶段也没有“立项文档是否完整”这一层检查。
  - 工具侧虽然有 PRD 模板与设计文档，但还没有把它接回真实初始化工作流。
- 结论:
  - 当前状态说明：PRD 在仓库设计里已经被认为重要，但还没有真正成为“项目初始化默认存在的文档”。
  - 后续应考虑至少做到其一：
    - `init` 默认生成 `PRD.md`
    - `doctor` / workflow gate 对缺失 PRD 给出 warning
    - 在 canonical workflow 中把“先确认 PRD”明确成写作前门禁

### 20. 已包裹实体校验链理论支持多类对象，但真实写作里 `@{}` 仍几乎只落在角色名上

- 现象:
  - 当前项目与现有实现已经默认把 `@{实体}` 视为重要对齐信号。
  - `doctor` 会检查章节中的已包裹实体是否已在 `entities/worldbook` 建档。
  - `story_review` 也已有 `wrappedEntitySignals`，说明设计目标并不只限于角色。
  - 但在《命灯照骨》的真实正文里，`@{}` 目前仍几乎只包裹角色名，势力、地点、特色物品等高价值对象很少被同样标记。
- 为什么这轮没解决:
  - 生成阶段虽然知道“建议使用 `@{实体}`”，但没有把“哪些对象必须优先包裹”写成足够明确的写中规则。
  - 自审阶段也没有专门检查“本章关键势力 / 关键地点 / 关键特殊物品是否已经进入包裹引用链”。
  - 工具侧当前更像“如果你已经包了，我就帮你校验”，而不是主动指出“这里本应包裹却没有包裹”。
- 结论:
  - 这说明当前 `@{}` 机制在产品层还没有真正成为“结构化写作协议”，而更像角色名辅助标记。
  - 后续需要考虑至少补齐其中一类能力：
    - 明确什么对象默认应包裹
    - 对章节内高频关键对象做“应包裹未包裹”提示
    - 区分角色、势力、地点、特殊物品等不同包裹类别的覆盖率

### 21. 缺少势力层级行为、地域扩张与战力扩张的结构化审查支撑

- 现象:
  - 当前 `worldbook.yaml` 已有：
    - `factions`
    - `locations`
    - `powerProgressions`
  - 但真实审查需求已经进一步指向：
    - 某势力若被定义为中级宗门，就应呈现出符合中层势力的行为边界，例如能压小宗门、又受更大势力调遣
    - 故事早期可能局限于一地，后期随着主角变强，地域、势力盘面与世界尺度应自然扩大
    - 战力上限、冲突等级、主角活动舞台，也应随阶段推进而变化
- 为什么这轮没解决:
  - 生成阶段虽然有 `worldbook` 真相层，但没有把“当前卷所在地域范围 / 当前战力天花板 / 当前主要势力层级关系 / 下一步扩圈方向”做成固定结构槽位。
  - 自审阶段没有一份稳定的“当前故事尺度”对照表，因此很难系统发现“势力行为不像它应有的层级”或“主角强度已经升级，但舞台和冲突还停在旧层级”。
  - 工具侧也没有把 `factions + locations + powerProgressions` 真正组装成面向写作审查的 judgement，导致这些真相层目前更多停留在静态存档，而不是动态约束。
- 结论:
  - 后续应考虑让协议和审查链显式暴露：
    - 当前卷/当前章所属地域层级
    - 当前主要势力盘面与上下位关系
    - 当前主角可承受的战力范围与风险上限
    - 预期的下一次世界扩张 / 地图升级 / 冲突升级节点
  - 否则即使 `worldbook` 已有基础字段，也很难稳定支撑长篇玄幻中最关键的“舞台扩张与力量升级”审查。

### 22. 缺少针对原创设定题材的“开篇世界观引导”结构槽位与门禁

- 现象:
  - 当前样稿审查表明：前 3 章未能充分讲清基础设定，也没有把世界观入口稳定建立起来。
  - 对常见题材，读者还能借既有类型经验自动补全；但像 `命灯` 这种较新颖的原创体系，读者无法默认理解其规则、代价和价值秩序。
- 为什么这轮没解决:
  - 生成阶段虽然已有 `project / worldbook / outline / context`，但没有把“前三章必须完成哪些基础世界观引导”做成显式结构任务。
  - 自审阶段也没有稳定检查“开篇读者此时是否已经理解最低必需设定集合”。
  - 工具侧当前更偏向检查设定是否存在、是否冲突，而不是检查“这些设定是否被读者顺利接收到”。
- 结论:
  - 后续需要考虑把原创设定题材的开篇 onboarding 纳入 repo-native 流程，例如显式记录：
    - 前三章必须解释到什么程度
    - 每章各自承担哪一段基础设定引导
    - 读者在第三章结束前应已理解哪些最低规则、代价与世界秩序
  - 否则“设定文件里有、正文也偶尔提到”并不等于读者真的已经被带进世界观。

### 23. 缺少“制度压迫成立条件”和“角色被困理由”的开篇结构约束

- 现象:
  - 当前开篇会让审查者自然追问：
    - 为什么底层弟子会去做 `压火` 这种高危工作
    - 这项工作到底是什么、对宗门有什么必要性
    - 宗门为何能长期用这种方式压榨他们
    - 这些人为何不脱离宗门
  - 这些问题并非后期深挖才需要，而是开篇处境能否成立的基础逻辑。
- 为什么这轮没解决:
  - 生成流程没有把“高危制度如何运转”“角色为什么离不开”做成前三章的固定说明任务。
  - 自审流程也没有单独检查“当前压迫结构是否已经具备最低自洽因果”。
  - 工具侧当前更关注设定字段是否存在、风险是否升级，还没有围绕“制度成立条件”形成显式门禁。
- 结论:
  - 后续需要考虑在开篇结构中显式记录：
    - 高危劳动/危险制度的作用
    - 底层角色参与它的被迫原因与现实收益
    - 退出成本、逃离障碍或替代路径为何不可行
  - 否则即使压迫氛围写出来了，读者也可能只觉得“不合理”而不是“残酷”。

### 24. 缺少关键反派/压迫方的人物逻辑建模与审查入口

- 现象:
  - 当前审查反馈表明，仅有“敌对角色在阻拦主角”并不足以形成高质量冲突。
  - 对手若没有自己的目标、判断、边界与人格，就容易显得蠢、工具化，像专门为主角服务的障碍物。
- 为什么这轮没解决:
  - 生成流程当前更擅长为主角提供外部压力，而不是同步构建“对手为什么这么做、他也在维护什么”。
  - 自审流程缺少针对关键反派/压迫方的固定检查表。
  - 工具侧当前没有一个清晰入口去表达“这个对手是否已从功能位成长为可信人物”。
- 结论:
  - 后续需要考虑在卷级或角色级结构中补充：
    - 关键敌对方的目标
    - 资源与约束
    - 常用手段与行为边界
    - 人格辨识度或魅力来源
  - 这样 review 才能审的不只是“反派有没有出手”，而是“他这样出手是否像一个真实的人/势力”。

### 25. 缺少对章节/提纲元信息泄漏进正文的硬性防线

- 现象:
  - 当前样稿出现了“第三章里压下灯库那场火”这类表述。
  - 这说明章节结构信息、回顾语言或工作流外层话语，有机会直接混入正常正文叙述。
- 为什么这轮没解决:
  - 生成流程没有把“作者侧结构语言不得进入正文”设成明确禁区。
  - 自审流程也没有专门扫“第几章 / 本卷 / 提纲 / 剧情安排”这类结构元信息是否误入正文。
  - 工具侧当前对 AI 味、方案文档腔、对照句式已有一些检测，但还没有把“非刻意 meta 语境下的章节层级元信息泄漏”视为高优先级硬伤。
- 结论:
  - 后续需要在 repo-native 规则里明确：
    - 正常正文默认禁止出现章、卷、提纲、剧情安排等作者侧结构语言
    - 若项目明确使用打破第四面墙或现实 `neta` 风格，才可例外放宽
  - 否则这类问题会持续以低频高伤害的形式出现，而且很容易穿过普通逻辑审查。

### 26. 缺少对“抽象递进 / 故作深沉腔”单句级 AI 味的更严格检验规则

- 现象:
  - 当前样稿出现了“心里竟没先起火。先起的是更清楚的一层算。”这类表达。
  - 这不是简单的词汇问题，而是人物反应被写成抽象概念递进，带出明显的 AI 造句痕迹。
- 为什么这轮没解决:
  - 生成流程没有把“人物反应优先具体化”设成足够强的写中约束。
  - 自审流程当前更偏重结构、设定、显性句式和元信息问题，对这种单句级抽象腔不够敏感。
  - 工具侧虽已有一部分 AI 味检测，但还缺少更严格的规则去抓：
    - 抽象概念层层递进
    - 伪深刻停顿句
    - 不落动作和感官的人物内心反应
  - 结论:
  - 后续需要把这类表达纳入更严格的 repo-native AI 检测规则，而不是仅靠人工读感发现。
  - 否则即使宏观结构没问题，正文仍会被大量单句级 AI 味拖坏。

### 27. 缺少对高频支架句组合的密度控制，而不是只做单句模式检测

- 现象:
  - 当前样稿反复出现一组高度相似的承重句：
    - `不是……是……`
    - `不是……不是……是……`
    - `像……`
    - `不是……更像……`
    - 短句切断后再递进一层
    - `不重/不轻/不偏/不倚` 这类刻意校准力度或姿态的句式
    - `真正……的，从来都是……` 这类价值判断定句
    - `还有什么？` 这类故意留半句、继续追问的短对话
  - 这些句式单独存在未必有错，但一旦在邻近段落或同章高频复用，就会形成非常明显的 AI 腔。
  - 这更接近一种风格习惯问题，而不是若干离散句式 bug；若继续按单句逐条补规则，只会越来越碎。
- 为什么这轮没解决:
  - 生成流程没有把这类句式当成“可用但应限频”的支架资源来管理。
  - 自审流程当前更偏向发现某一句像 AI，而不是发现“同一类句法骨架已经在整章里复用过密”。
  - 工具侧也还缺少组合式频率检测，导致即使单句都不算硬伤，累计起来依然会让正文发假。
- 结论:
  - 后续需要把 AI 检测从“单条句式命中”进一步收敛到：
    - 句式类别统计
    - 邻近段落重复密度
    - 组合式支架复用告警
    - 角色口吻归属检查，避免把本应属于某一人物的说话习惯扩散成全员通用模板
    - 同类样例统一归并，而不是每出现一个新变体就新增一条独立规则
  - 否则规则会漏掉大量“单句勉强成立、通篇看极不自然”的问题。

### 28. 缺少首卷引入篇的复杂度预算与回收节奏门禁

- 现象:
  - 当前样稿在第 1 章到第 10 章之间持续增加伏笔、疑云、势力、人物和特殊物品，但已完成的回收点偏少。
  - 这对长篇后续展开未必是坏事，但对“第一卷作为引入篇”来说，复杂度已经偏高。
- 为什么这轮没解决:
  - 生成流程没有把“首卷只能承载多少核心谜题、多少核心对象、多少未回收暗线”做成明确预算。
  - 自审流程也没有追问“这一卷是在稳步带读者入场，还是已经把理解成本抬得过高”。
  - 工具侧虽有伏笔账本、实体清单和 worldbook，但还没有把这些信息汇总成“当前卷复杂度是否超标”的门禁判断。
- 结论:
  - 后续需要考虑为首卷或引入篇增加 repo-native 约束，例如：
    - 核心未回收疑点数量上限
    - 新势力 / 新人物 / 新特殊物品引入节奏
    - 短期回收点分布要求
  - 否则系统会默认“多就是深”，但读者实际感受到的往往只是“乱和累”。

### 29. 缺少对 POV 可见性 / 可知性边界的硬性审查

- 现象:
  - 当前样稿会写出类似“正落在沈照右后方……边沿”这类信息。
  - 若正文当时是紧贴主角视角，这类后方精确落点信息并不属于角色可直接感知范围。
- 为什么这轮没解决:
  - 生成流程没有把“当前信息是谁看见的、谁知道的”作为逐句级约束。
  - 自审流程也没有单独扫 POV 越界，因此镜头调度很容易压过人物感知边界。
  - 工具侧当前缺少一类专门规则去检查：
    - 方位越界
    - 感知越界
    - 未显式切视角却写入他视角信息
- 结论:
  - 后续需要把 POV 边界纳入 repo-native 审查规则，至少明确：
    - 紧贴视角下允许写什么
    - 哪些信息只能通过反应、声音、余光、推断来表达
    - 什么时候必须显式切换视角或改成客观镜头
  - 否则正文很容易为了制造画面感而牺牲最基本的视角逻辑。

### 30. 缺少“核心概念最晚解释时点”的卷级门禁

- 现象:
  - 当前样稿到第 13 章仍让读者难以说清 `灯` 与 `人` 的基本关系。
  - 这说明首卷没有为“核心原创概念必须在何时前被读者理解到什么程度”设定明确门槛。
- 为什么这轮没解决:
  - 生成流程没有把“核心概念 onboarding”拆成按章交付的结构任务。
  - 自审流程也没有在首卷中段主动追问“核心概念是不是还停留在纯神秘状态”。
  - 工具侧当前更偏向记录设定和审冲突，而不是审“设定是否已经被读者真正理解”。
- 结论:
  - 后续需要在 repo-native 流程中显式定义：
    - 哪些概念是项目级核心概念
    - 每个核心概念最晚应在第几章前完成最低解释
    - 到该节点时读者应已理解的最低内容
  - 否则原创设定作品很容易一路保密到读者放弃，而不是保密到读者更想追。

### 31. 缺少按题材区分的“首卷疑点预算”与“解释节点预算”

- 现象:
  - 当前样稿在引入卷前半段持续追加伏笔和疑云，但没有足够阶段性回收。
  - 这类铺法更像悬疑解密题材的高密度问题堆叠，而当前项目主承诺显然不只是纯悬疑。
- 为什么这轮没解决:
  - 生成流程没有按题材区分“可以留多少谜”“什么时候必须解释多少”。
  - 自审流程也没有把“疑点预算”与“解释预算”作为首卷 gate 的一部分。
  - 工具侧虽已开始关注复杂度和伏笔，但还没有把“错误题材策略下的疑点过载”转成明确 judgement。
- 结论:
  - 后续需要考虑为首卷引入卷增加按题材分层的预算约束，例如：
    - 未回收核心疑点数量
    - 必须在中前段完成的解释节点数量
    - 可延后到中段/高潮再解的谜题类型
  - 否则系统会默认“疑点越多越勾人”，但真实效果可能是“越多越劝退”。

### 32. 缺少“已知实体自动包裹建议 + 缺失实体一次性反馈 + 章节实体摘要注入”的审查闭环

- 现象:
  - 当前正文中已经建档的实体名，仍会出现部分地方包裹、部分地方未包裹的情况，例如同一章内出现：
    - `@{岳怀川}`
    - `岳怀川`
  - 这会让实体对齐、设定核对和后续审查链都变得不稳定。
  - 同时，若正文出现了尚未建档但明显应视为实体的名称，当前流程也缺少足够好的“一次性提醒 + 补档建议”。
- 为什么这轮没解决:
  - 生成流程没有在写后或审查前自动对照“本项目已知实体表”，去提示哪些 mention 应包裹、哪些新 mention 需要建档。
  - 自审流程也没有把“本章涉及的实体设定摘要”稳定带给 AI，导致 AI 仍常退回原始文件里自己查。
  - 工具侧当前虽然已有 `wrappedEntitySignals` 和 `doctor` 标签合法性检查，但还缺少更实用的一整套闭环：
    - 对已知实体给出自动包裹建议
    - 对缺失实体给出补档提示
    - 对同一实体去重，避免在一章里反复重复反馈
    - 在审查时只注入当前章相关的实体设定切片，便于核对
- 规则例外补充:
  - 对话引号中的称呼不应默认强制包裹。
  - 原因是人物在他人口中可能使用：
    - 别名
    - 临时称呼
    - 职位称呼
    - 带情绪色彩的叫法
  - 这些称呼本身可能变化，也未必适合直接收敛成单一 canon 名称。
- 结论:
  - 后续需要考虑把这条链收敛成 repo-native 能力，例如：
    - 章节 mention 扫描后，对已建档实体输出“建议包裹列表”
    - 对未建档但高置信的实体 mention 输出“建议建档列表”
    - 同一实体在同一章内只反馈一次
    - `review` / `doctor` / 自审 prompt 只携带当前章相关实体的最小设定摘要，而不是让 AI 自己再翻全量文件
    - 对引号内 mention 默认降级处理，优先忽略或只做低置信提示，不直接要求补 `@{}`
  - 否则实体协议虽然存在，AI 在真实写作与审查里仍会反复退回手工读文件和手工补标签。

### 33. 缺少“卷 = 一个小故事单元”的闭环 gate，导致高潮不等于卷完成

- 现象:
  - 当前样稿到 `chapter-023` 虽然已有明显高潮段，但人工审查仍认为不足以构成第一卷完整收尾。
  - 这说明现有流程更容易把“局势升级、追兵逼近、高潮形成”误当成“卷已经成立”。
- 为什么这轮没解决:
  - 生成流程没有把“本卷需要收束哪个核心问题、交出什么阶段性结果”做成明确 gate。
  - 自审流程此前也缺少对“卷是否完成一个相对独立小故事”的强约束。
  - 工具侧当前虽已开始建设卷级自审，但还没有把“高潮 != 闭环”作为清晰判断规则。
- 结论:
  - 后续需要在 repo-native 卷级 gate 中显式检查：
    - 本卷主问题是否阶段性收束
    - 本卷是否给出一次完整小故事交付
    - 卷尾是否既有余波，又不至于完全没有完成感
  - 否则系统会高估“紧张感”和“完成度”的等价关系。

### 34. 缺少对“爽点引擎 / 目标人群 / 核心体验”的卷级结构化约束

- 现象:
  - 当前人工审查难以回答：
    - 这本书给读者的核心体验是什么
    - 它的主要爽点是什么
    - 它更适合什么读者
  - 这说明商业定位虽然可能写在项目层，但尚未被稳定转译到卷级执行与审查中。
- 为什么这轮没解决:
  - 生成流程没有把“本卷主爽点、压抑来源、爆发时刻、目标人群预期”做成显式执行槽位。
  - 自审流程也没有稳定核查“本卷是否至少兑现过一次主承诺”。
  - 工具侧当前对商业对齐仍偏启发式，没有围绕“爽点兑现”形成明确门禁。
- 结论:
  - 后续需要在 repo-native 流程中把以下信息显式化并进入 review：
    - 目标读者
    - 核心体验
    - 主爽点引擎
    - 本卷压抑/爆发分布
    - 本卷至少一次明确兑现节点
  - 否则作品可能长期只会“压”，却不知道何时该“扬”。

### 35. 缺少卷级结构模板与经典叙事节奏对照能力

- 现象:
  - 当前首卷没有稳定体现出引入卷应有的结构职责：建立基本世界观、抓住开篇兴趣、展开一个相对完整的小单元剧情。
  - 从人工审查视角看，现有工作流没有明显帮助 AI 对照三幕式、五幕式或等价结构来检查卷级节奏。
- 为什么这轮没解决:
  - 生成流程当前更多依赖自由文本大纲、beats 和 scenePlans，没有明确的卷级结构模板。
  - 自审流程也没有把“结构职责是否完成”转成固定问卷或门禁。
  - 工具侧缺少面向卷级的结构审查稿，无法帮助 AI 或人工快速判断“这一卷现在卡在哪一幕、缺哪一段职责”。
- 结论:
  - 后续需要考虑补充 repo-native 卷级结构支持，例如：
    - 三幕式 / 五幕式模板槽位
    - 引入卷专用结构检查表
    - 卷级结构完成度审查
  - 否则 AI 容易持续堆材料，却缺少宏观结构的收束框架。
