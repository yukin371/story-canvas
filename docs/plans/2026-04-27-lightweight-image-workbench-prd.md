# Story Canvas 轻量生图工作台 PRD

> 最后更新: 2026-04-27
> 状态: 开发中
> 关联当前入口: `docs/roadmap.md`

## 1. 背景

当前仓库已经具备：

- `text-to-image`
- `image-to-image`
- provider 配置
- dry-run request 预览

但现阶段生图链路仍偏“命令能力 + 参数暴露”，对普通用户不够友好。要把它收敛成可持续维护的产品子系统，需要先明确一个更轻的产品方向：

- 用户不需要先理解复杂提示词工程
- 系统先提供内置模板，再接受用户追加提示词
- 常见能力覆盖即可，不做重型图像工作站

同时，这套能力未来可能从 `Story Canvas` 中分离出来，成为独立在线生图工具，因此现在就要定义好“通用能力层”和“小说场景增强层”的边界。

## 2. 产品定义

| 字段 | 内容 |
|------|------|
| 产品名 | Story Canvas Image Workbench |
| 当前形态 | Story Canvas UI 内嵌工作台 |
| 未来形态 | 可拆分为独立在线生图工具 |
| 当前主要用户 | 需要快速出角色图、章节图、宣传图的作者/操作者 |
| 未来用户扩展 | 普通生图用户、商业化生图用户 |

## 3. 产品目标

1. 降低第一次使用生图功能的理解门槛。
2. 通过“模板 + 用户补充提示词”形成稳定的轻量提示词系统。
3. 覆盖最常见的四类基础能力：
   - 文生图
   - 图生图
   - 批量生图
   - 重绘 / 局部重绘
4. 保持产品轻量，不演化成节点式复杂工作流编辑器。
5. 为未来拆分成独立在线工具预留普通版 / 商业版产品分层。

## 4. 非目标

- 不做 ComfyUI 式可视化节点编排
- 不做复杂图层编辑器
- 不做大型素材管理 DAM
- 不在本阶段支持多人协作审片流
- 不把全部高级参数直接暴露给新手用户

## 5. 核心用户问题

### 5.1 当前问题

- 用户需要自己构造 prompt，门槛高
- 常用风格、镜头、构图、角色设定缺少内置模板
- 批量生成与重绘等常见能力没有形成统一产品心智
- 如果未来拆出成在线产品，当前能力边界不够清晰

### 5.2 目标体验

- 用户先选目标和模板，再补一句额外提示词就能出图
- 批量生成、图生图、重绘都在同一工作台中完成
- 默认路径简单，进阶控制逐步展开
- 即使以后脱离小说协议，也能保留一套通用生图产品骨架

## 6. 产品原则

### 6.1 默认简化

- 默认只展示用户最常需要的字段：
  - 目标
  - 生成模式
  - 模板
  - 额外提示词
  - 参考图 / mask
  - 数量

### 6.2 模板优先

- 系统必须内置模板，用户不从空白 prompt 开始
- 模板可按用途分层：
  - 角色设定图
  - 章节场景图
  - 宣传图 / 封面概念图
  - 商业图 / 电商图 / 品牌图（未来在线工具）

### 6.3 轻量进阶

- 允许高级参数，但默认折叠
- 不鼓励一次性暴露大量 provider 原始参数

### 6.4 可拆分

- 业务增强层（小说章节、角色、项目默认值）与通用生成层（模板、提示词、模式、输出）必须可拆

## 7. 功能范围

### 7.1 In Scope

- 文生图
- 图生图
- 批量生图
- 重绘 / 局部重绘
- 模板驱动 prompt 系统
- 用户额外提示词追加
- 生成历史与重跑
- 普通版 / 商业版的预留产品分层

### 7.2 Out of Scope

- 工作流节点编辑器
- 精修级图像编辑器
- 模型训练 / LoRA 训练平台
- 高度自由的资产审批系统

## 8. 提示词系统设计

### 8.1 目标

提示词系统的目标不是让每个用户都学会 prompt engineering，而是把提示词能力产品化。

### 8.2 基本结构

每次生成由四层组成：

1. `system template`
   - 系统内置模板
   - 决定画面结构、用途和基础质量要求
2. `style modifiers`
   - 风格、镜头、光线、构图、材质等可选修饰层
3. `user extra prompt`
   - 用户额外补充的自由文本
4. `negative / safety / commercial policy`
   - 负向约束、平台安全约束、商业输出规范

### 8.3 模板分层

- 按用途：
  - character
  - chapter-scene
  - cover-concept
  - promo
  - product / marketing（未来）
- 按风格：
  - 写实
  - 动漫
  - 轻小说
  - 国风
  - 商业广告（未来）
- 按复杂度：
  - quick
  - standard
  - detailed

### 8.4 用户输入策略

- 用户默认只需要补充一句自然语言描述
- 进阶用户可选追加：
  - 风格词
  - 构图词
  - 角色特征词
  - 品牌/商业限制词

### 8.5 多样化要求

- 模板库不能只提供单一审美方向
- 至少覆盖：
  - 写实人物
  - 动漫人物
  - 叙事场景
  - 海报式宣传图
  - 商业展示图（为未来在线工具预留）

## 9. 交互设计

### 9.1 默认主流程

1. 选择目标
2. 选择模式
3. 选择模板
4. 填写额外提示词
5. 选择数量
6. 执行 dry-run / generate

### 9.2 模式支持

- 文生图
  - 适合角色设定图、场景图、概念图
- 图生图
  - 适合已有设定稿或历史图二次生成
- 批量生图
  - 支持一次多张输出，用于快速挑选
- 重绘 / 局部重绘
  - 支持基于 mask 的局部修改

### 9.3 复杂度控制

- 不把所有参数平铺在首屏
- 首屏只显示核心操作
- 高级设置折叠在二级区

## 10. 普通版与商业版预留

### 10.1 普通版

- 通用风格模板
- 基础文生图 / 图生图 / 批量 / 重绘
- 个人项目历史复用

### 10.2 商业版

- 商业模板包
- 品牌一致性约束
- 输出规格约束
- 商用安全 / 合规提示
- 更强的批量复用和团队交付能力

## 11. Story Canvas 内嵌态与未来独立态

### 11.1 内嵌态

- 优先消费章节、角色、项目默认值
- 强化小说相关模板

### 11.2 独立态

- 去掉小说协议依赖后，保留模板系统、模式系统、历史系统和商业分层
- `Story Canvas` 可视为其垂直场景版本

## 12. 验收标准

1. 新手用户无需自己从零写 prompt。
2. 系统内置模板能覆盖至少角色图、场景图、宣传图三类主路径。
3. 界面明确支持文生图、图生图、批量生图、重绘四类能力。
4. 产品边界保持轻量，没有滑向重型工作流编辑器。
5. 文档中已经为未来独立在线工具的普通版 / 商业版分层留出结构。

## 13. 风险

- 模板过少会让“降低门槛”落空
- 模板过多又会让轻量产品变成复杂配置台
- 若现在就把 Story Canvas 业务字段和生图底层强耦合，未来拆分成本会很高
