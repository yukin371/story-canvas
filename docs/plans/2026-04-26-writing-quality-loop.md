# 写作质量三段式闭环

## 1. 背景

在《归墟》的人工审查过程中，已经暴露出一个很明确的问题：

- 模型会“贴合大纲/细纲”，但不等于会自然写成小说。
- 当约束不够细时，容易出现：
  - 章间过渡断裂
  - 情节像在执行提纲而不是自然展开
  - 角色言行与设定、压力、关系脱节
  - 明显的“AI 味”、结构化归纳腔、程式化过渡句

这和 AI 编程很像：

- 不能假设模型天然可靠
- 必须前置约束
- 必须过程纠偏
- 必须产后检测与循环修复

## 2. 目标

把写作质量闭环明确拆成 3 个阶段：

1. 开始动笔前知晓规则
2. 动笔时遵循基本规则
3. 完成后使用 CLI 规则进行检测、评分、自审与修正

并把质量改进方向稳定收敛到以下几类：

- 设定遵循
- 故事与背景逻辑
- 章间衔接与场景过渡
- 角色塑造与关系推进
- 行文风格与 AI 味控制
- CLI 检测 / 规则评分 / 循环修复

## 3. 非目标

- 不把“AI 完全自动生成小说”当成近期目标
- 不用一次性大 prompt 解决全部质量问题
- 不把所有问题都误判成纯风格问题
- 不把人工审查完全替换成 CLI 输出

## 4. 三段式闭环

### 4.1 开始动笔前知晓规则

开写前，模型必须先知道当前章真正受哪些约束，而不是只看大纲标题。

最小前置约束至少应包括：

- 当前卷 / 当前章在整条长线中的位置
- 上一章的结果、当前章的承接点、下一章需要交出的推进
- 当前章必须延续的世界规则、人物状态、关系张力、伏笔窗口
- 本题材的风格边界与明显禁忌
- 本章允许的解释密度、信息投放方式和节奏目标

核心判断：

- 不是“这一章要写什么”
- 而是“这一章为什么必须从上一章自然长出来”

### 4.2 动笔时遵循基本规则

写作过程中，至少要约束以下基本规则：

1. 不能只按大纲点对点执行，必须保证段落、场景、章节间有自然过渡。
2. 不能为了推进提纲强行切换情绪、立场、动机和关系状态。
3. 不能让人物只承担剧情功能，必须维持人物辨识度和稳定反应逻辑。
4. 不能用“目标/风险/约束/时间窗口”这类方案化表达直接替代叙事。
5. 不能高频使用程式化过渡句、说明腔、总结腔和 AI 套话。

同时要明确：

- 细纲是执行路径，不是最高真相源。
- 当细纲安排与世界规则、既成后果、角色当前状态和合理反应冲突时，应优先守住后者，再调整具体 scene path。
- 但“人物自然发展”不能成为随意跑偏的借口；若偏离细纲，必须仍然完成本章应交出的推进，并在写后进入 review / 自审 / 修正闭环。

其中，章间衔接应单独看待：

- 当前章开头要能接住上一章末尾留下的情绪、信息或行动后果
- 当前章结尾要为下一章留下明确可承接的状态，而不是只结束一个提纲事项

### 4.3 完成后 CLI 检测、自审与修正

成稿后不能直接接受结果，至少要进入：

1. `style`
2. `consistency`
3. `review chapter` / `review scene`
4. 必要时 `doctor`

并执行一轮整体评分自审：

- CLI 发现了什么
- CLI 没发现但人工读感明显有问题的是什么
- 哪些属于误报
- 哪些属于漏报
- 哪些必须通过重写章节、补桥接段、调整前后章状态来修

原则：

- 不是“跑完命令就算过”
- 而是“检测 -> 自审 -> 修正 -> 复检”形成闭环

## 5. 当前最需要补强的问题

### 5.1 章间衔接与过渡

《归墟》已出现第 12 章与第 13 章之间几乎无过渡衔接的问题。

这类问题说明当前系统还缺：

- 对“上一章结果 -> 下一章起点”的显式检查
- 对“像贴细纲而不是自然续写”的人工审查维度
- 对章节桥接质量的 review 口径

### 5.2 AI 味过重

当前“AI 味”不只是词汇问题，也包括：

- 方案文档腔
- 程式化过渡句
- 段落组织过于整齐而缺少叙事自然性
- 解释替代表演
- 为了完成提纲事项而非为了故事推进而写

### 5.3 整体评分后缺少修复循环

当前系统已经有：

- `style`
- `consistency`
- `review`

但还需要更明确的使用原则：

- 它们不是单次打分器
- 而是修正循环的输入

## 6. 对 v1.0 的实际要求

`v1.0` 不应被理解成“已经能一键稳定写好小说”，而应被理解成：

- 已经具备明确的写前约束机制
- 已经有基础的写中规则边界
- 已经能在写后用 CLI 给出有效检测和修正方向
- 已经能通过多风格、多卷级人工审查持续回灌缺陷

## 7. 建议后续实现方向

### 7.1 写前

- 让 `context refresh` / outline / review 产物更明确暴露：
  - 上章结果
  - 本章承接点
  - 本章必须维持的状态与禁忌

### 7.2 写中

- 形成一套可直接喂给模型的“基本写作规则”约束模板
- 尤其强调：
  - 章间过渡
  - 角色稳定性
  - 非方案腔表达
  - 说明与表演的平衡
  - 细纲可调整，但必须说明哪些是 `must_keep`，哪些只是 `flexible_path`

### 7.3 写后

- 强化 review 对“衔接 / 过渡 / 承接上一章结果”的评分提示
- 强化 style 对“AI 味”的检测
- 把 CLI 输出明确转成修正动作，而不是只返回抽象分数

## 8. 验证口径

后续真实样例人工审查至少要补问：

1. 当前章是否自然承接上一章
2. 当前章是否只是把大纲事项写成正文
3. 是否存在明显 AI 味或结构化归纳腔
4. CLI 是否提前发现了这些问题
5. 若 CLI 未发现，应归类为 `style / consistency / review / workflow` 哪一侧缺口

## 9. 当前结论

对《归墟》这类复杂长篇，真正需要的不是更大的生成 prompt，而是更硬的质量闭环：

- 写前知道规则
- 写中遵守基本规则
- 写后检测、评分、自审、修复、复检

这应作为 `v1.0` 写作质量提升的主方向，而不是把 AI 当成天然可靠的黑盒作者。
