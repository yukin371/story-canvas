# Story Canvas 路线图

> 最后更新: 2026-04-27
> 状态: 当前唯一执行入口

## 1. 当前发布目标

- `v1.0`: 第一个整体发布版本，目标是把 Story Canvas 从“早期协议 / CLI 原型”收敛成“完整、有效的故事与视觉工作流产品”

## 2. v1.0 发布范围

### 2.1 核心定位

- `v1.0` 聚焦三件事：
  - 故事协议稳定
  - workflow 闭环可验证
  - 样例矩阵可支撑回归与对外说明
- `v1.0.x` 的发布判断以“故事工程管理能力”是否成立为主，但图片能力与早期 UI 已进入并行开发，不再完全后置

### 2.2 故事工程协议

- 项目可按代码仓库思路管理：结构清晰、协议稳定、状态可追溯、可迁移
- 核心管理域完整且可协同：
  - `project.yaml`
  - `outline`
  - `worldbook` / 世界约束
  - `entities`
  - `foreshadowing`
  - `threads`
  - `timeline`
  - `reviews`
  - `projections`
  - `workflow`

### 2.3 工作流能力

- 支持显式 workflow 状态推进
- 支持 gate 决策记录、恢复、导出
- 工作流与故事协议一致，不依赖隐式人工记忆
- 对写作质量的工作流闭环，必须逐步收敛成：
  - 开写前知晓规则
  - 写作时遵循基本规则
  - 完成后通过 CLI 检测、评分、自审与修正
- 工作流闭环必须区分两个层级：
  - `章节闭环`: 单章完成后进入检测、自审、修正、复检
  - `卷级闭环`: 一个完整卷/小故事单元完成后，先进行 AI 卷级自审与修正，再进入人工审查

### 2.4 样例与发布基线

- 至少支持短篇与长篇两个真实样例闭环
- 文档能说明各样例验证了什么
- CLI 主闭环可离线验证，不依赖外部服务才能工作

### 2.5 产品边界与命名迁移

- 产品名称已收敛为 `Story Canvas`
- 仓库名与安装分发名收敛为 `story-canvas`
- 主 CLI 命令为 `story-canvas`，主 Python 模块入口为 `story_canvas`
- `story-harness` CLI 与 `story_harness_cli` Python 模块暂时保留为兼容别名，避免在 `v1.0.x` 阶段引入不必要的兼容性抖动

## 3. v1.0 收口清单

### Track 1: 协议收敛为产品级故事管理能力

- 目标: 从“角色 + 大纲 + 章节”升级到“可维护的故事真相层与动态状态层”
- 当前重点:
  - 落稳 `project` / `outline` / `entities` / `foreshadowing` / `threads` / `timeline`
  - 引入更完整的世界/背景/规则约束协议
  - 让角色状态变化和伏笔回收具备追溯能力
- 验收标志:
  - 长篇样例可证明协议不是摆设
  - `doctor` / `outline check` / `context refresh` / `review` 能消费这些协议

### Track 2: 工作流闭环稳定化

- 目标: 把当前写作流程变成可审计、可恢复、可导出的显式状态机
- 当前重点:
  - `workflow status/run/advance/reset/export`
  - gate 结果持久化
  - 与 review / projection / context 的闭环衔接
  - 让“写前约束 -> 写中遵循 -> 写后检测与修复”成为真实可执行的质量闭环，而不是只停留在大纲推进
  - 在当前单章 gate 之外，补上“整卷 AI 自审 -> 修正 -> 复检 -> 人工审查”的卷级闭环
- 当前进度:
  - `review-rules.yaml` 已进入真实检测链，`metaLeakage` 与 `povOverreach` 已可在 `style/review` 中统一输出 judgement，并支持白名单式豁免
  - 首批中文高频 AI 风格规则已落到 `style` 主链：`contrastFlipPattern`、`analogicalPivotPattern`、`templateCatchphrasePattern`、`paragraphReadability`
  - `clusteredAIPhrasing` 已接入 `style/review` 主链，可把多项轻度 AI 句式与可读性信号聚成统一风险，减少“单项都不重但整体 AI 味偏高”的漏报
  - 新规则已接入 `style check`、`review chapter`、`review scene` 与样例回归，不再只停留在人工口头约束
  - 长篇样例基线已开始受新规则影响，说明“AI 味控制”已从提示信号进入真实质量门槛
- 验收标志:
  - 至少一个真实项目跑通规划 -> 写作 -> 审查 -> 投影 -> 导出
  - 系统能明确区分“章节已闭环”和“整卷已闭环”，避免把单章通过误判成可进入人工终审
  - 人工审查默认发生在一个完整卷/小故事单元完成，并经过一轮 AI 卷级自审之后
  - 人工审查确认系统不只是“贴合大纲”，而是能持续改善设定遵循、逻辑、衔接、角色塑造、风格与 AI 味控制

### Track 3: 发布基线与样例矩阵

- 目标: 让 `v1.0` 有可对外展示和回归验证的样例基础
- 当前重点:
  - 固化短篇回归样例
  - 固化长篇商业样例
  - 补一个“复杂长篇暗线”样例视角，优先参考 `demo-guixu`
  - 固化 `玄幻 / 西幻 / 日轻 / 言情` 四类典型风格测试矩阵，而不是只验证单一题材
  - 每类风格至少准备 `2` 卷级别的真实写作测试；更优目标是其中至少 `2` 类达到 `3` 卷
  - 第一卷重点验证“黄金三章”能力：是否能快速立住钩子、承诺、主角处境与继续阅读欲望
  - 第二卷开始重点验证故事展开质量：世界/背景逻辑、伏笔延续、人物塑造与关系推进是否稳得住
  - 把“章间衔接是否自然”“是否只是把提纲写成正文”“AI 味是否过重”纳入固定人工审查维度
  - 在人工审查前，固定要求 AI 先输出卷级缺陷清单、归因和修正结论，而不是直接把原稿交给人
- 验收标志:
  - 至少一短一长稳定通过回归
  - 四类典型风格都有固定测试项目，不再只靠临时样例
  - 最低达到“四类风格各有两卷级人工审查样例”的发布前验证基线
  - 文档能说明各样例验证了什么、发现了什么缺陷、后续如何回灌修复
  - 至少一个复杂长篇样例证明系统能支持“检测 -> 自审 -> 修正 -> 复检”的质量循环，而不是一次性出稿
  - 至少一个复杂长篇样例证明系统能在卷级别识别“生成漏做 / 自审漏看 / 工具漏报”这三类问题来源

### Track 4: 发布级文档与口径收敛

- 目标: 让路线图、验收文档、README 和真实状态保持一致
- 当前重点:
  - 区分 `v1.0` 已承诺能力与 `v1.1+` 延后能力
  - 固化离线运行、验证、迁移和已知限制口径
  - 避免文档把实验能力误写成发布阻塞项
- 验收标志:
  - `docs/roadmap.md`、发布验收文档和 `README.md` 没有明显冲突
  - 发布范围、阻塞项、残留风险可被明确陈述

### Track 5: 生图能力并行推进

- 目标: 让插画生成从边缘扩展能力收敛成可持续维护的产品子系统
- 当前重点:
  - 稳定 `text-to-image` / `image-to-image`
  - 默认支持官方 OpenAI，兼容网关通过 `base_url` 覆盖
  - 记录 prompt、revised prompt、response id、provider request、输出资产路径
- 当前进度:
  - 插画 provider 已切到 `Responses API + image_generation`
  - `illustration generate` 已支持 `--base-url` 与 `--response-model`
  - 官方 OpenAI key 与兼容网关 key 均可走同一套 OpenAI 风格接口
  - dry-run 已可验证 provider request、输出路径与模式切换
  - 本地 UI API 已接入真实 generate，可落盘资产并刷新项目 summary
- 下一方向:
  - 增加 provider 配置可视化与资产预览
  - 补充历史任务重跑、批量参数与更多模板复用
  - 补足真实项目的文生图 / 图生图回归样例，而不只停在 smoke test
- 验收标志:
  - 至少一个真实项目能稳定生成并记录插画资产
  - provider 行为可通过 dry-run 与 smoke test 验证

### Track 6: 早期 UI 并行推进

- 目标: 在不改变文件协议真相源的前提下，优先降低小说生成测试的人力成本
- 当前重点:
  - 项目状态浏览
  - 章节与审查结果浏览
  - 插画生成与结果预览
  - provider 配置与历史复用
- 当前进度:
  - 已建立 `Vue 3 + Vite + TDesign` 骨架
  - 当前本地联调端口固定为 `43187`，配套 API 端口为 `43188`
  - 项目列表、章节/审查概览、插画工作台已从 mock data 切到真实项目 API
  - 插画面板已支持文生图 / 图生图 / 重绘的真实 generate、dry-run、按项目默认配置回填与历史复用
- 下一方向:
  - 增加任务状态回刷后的资产预览
  - 增加 review packet / context lens / change request 浏览面板
  - 决定 `v1.0.x` 是否附带“本地 UI 预览模式”，还是继续以实验能力交付
- 验收标志:
  - 最常见的人肉测试步骤不再全部依赖反复敲 CLI 和手动翻文件
  - UI 只作为协议可视化壳，不引入新的真相源

## 4. 当前阻塞

- 本地 `pip install -e .` 仍可能受镜像/证书问题影响
  - 影响: 无法稳定通过已安装命令入口验证
  - 临时路径: 使用 `PYTHONPATH=src python -m story_canvas`；旧路径 `python -m story_harness_cli` 仍兼容
- 当前工作树存在大量并行改动
  - 影响: `v1.0` 收口前需要避免路线图与真实代码状态长期分叉

## 5. v1.0 发布判断标准

- 核心协议稳定，旧项目可兼容加载或有明确迁移路径
- CLI 主闭环可离线验证，不依赖外部服务才能工作
- 工作流能力和故事管理能力不是分离孤岛
- 至少一个长篇项目证明工具不是只适合玩具样例
- 发布前至少完成一轮多风格、多卷级真实写作验证，并经过人工审查回灌缺陷
- 发布前至少证明系统能围绕设定遵循、逻辑、衔接、角色塑造、行文风格、AI 检测、规则评分与循环修复形成可执行闭环
- 发布前至少证明：单章闭环通过不等于可人工终审，卷级闭环必须包含“AI 自审 -> 修正 -> 复检 -> 人工审查”
- 文档已明确区分发布能力、非目标与已知限制

## 6. v1.1 展望

- `v1.1`: Story Canvas 的第一版更完整公开视觉界面
- 目标:
  - 提供阅读、审查、简单修改结构化产物的界面
  - 支持查看项目状态、workflow、review、伏笔、角色状态、插图资产
  - 承接并强化已经提前启动的图片与审查 UI 轨道，而不是从零开始做 UI
- 边界:
  - Web UI 只是文件协议的可视化壳，不应成为新的真相源
  - 生图能力在 `v1.1` 以“可视化管理”优先，不追求复杂图像编辑器
  - 不在这一版追求多人协作系统

## 7. v1.2 展望

- `v1.2`: 模板化与自由编排增强
- 目标:
  - 丰富题材模板
  - 管理 prompt packs / 提示词资源
  - 支持 workflow 自由编排
  - 提供默认模板与参考项目
- 前提:
  - `v1.0` 协议稳定
  - `v1.1` UI 已证明对协议只是外壳而非替代

## 8. 历史背景

- `v0.2` 路线中的已完成项仍然有效，构成当前 `v1.0` 的技术起点：
  - timeline / search / causality
  - review chapter / review scene
  - scenePlans 维护闭环
  - positioning / storyContract / commercialPositioning
  - export 多格式、entity graph、样例矩阵
- 插图命令与 provider 基础设施仍保留为既有能力，但不再作为 `v1.0` 发布硬阻塞
- 这些能力不再单独作为执行入口，而作为当前路线图的技术基础
