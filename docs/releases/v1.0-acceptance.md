# v1.0 发布验收

> 最后更新: 2026-04-26
> 状态: 草案
> 适用版本: `v1.0`
> 关联: [当前路线图](../roadmap.md), [故事约束与情绪协议设计](../plans/2026-04-25-story-constraint-protocol.md)

## 1. 发布目标

`v1.0` 不是“再多一批命令”，而是第一个可对外说明为“完整、有效的 Story Canvas 核心版本”的发布。

发布通过的前提是：

- 项目已经可以像代码仓库一样管理
- 核心故事协议稳定且可验证
- 工作流能力、审查能力、投影能力不是孤立功能
- 样例矩阵与文档口径足以支撑对外发布
- 生图能力与早期 UI 即使并行推进，也没有破坏 `v1.0.x` 的核心边界

## 2. Must Have

### 2.1 故事工程协议

- [ ] `project.yaml` 具备稳定的作品级协议
- [ ] `outline` / `entities` / `foreshadowing` / `threads` / `timeline` / `reviews` / `projections` / `workflow` 已形成稳定状态域
- [ ] 长篇所需的世界/背景/规则约束有正式协议入口
- [ ] 旧项目可兼容加载，或有明确迁移路径

### 2.2 核心 CLI 闭环

- [ ] 规划 -> 写作 -> 分析 -> 评审 -> 投影 -> 上下文刷新 -> 导出 能跑通
- [ ] `doctor` 能对核心协议做结构化检查
- [ ] `outline check` 能充当写前门禁
- [ ] `workflow` 能显式记录 gate 决策与阶段推进
- [ ] 核心闭环在无外部服务时仍可运行

### 2.3 样例与验证

- [ ] 至少一个短篇样例可作为回归基线
- [ ] 至少一个长篇样例可作为商业长篇基线
- [ ] 至少一个复杂长篇/暗线样例可验证更重的故事协议
- [ ] 样例文档明确写出“这个样例验证了什么”

### 2.4 发布级治理

- [ ] `docs/roadmap.md` 与真实状态一致
- [ ] 模块文档与关键 guide 没有明显过期
- [ ] `README.md` 至少能说明当前产品边界与运行方式
- [ ] 发布范围、阻塞项、残留风险可被明确陈述

### 2.5 Story Canvas 迁移口径

- [ ] 产品名称、README、路线图、验收文档的口径一致
- [ ] 仓库命名迁移完成后，`story-canvas` 成为主入口，`story-harness` 仍保持兼容
- [ ] 对“官方 OpenAI 默认、兼容网关可选覆盖”的 provider 行为有明确文档说明

## 3. Should Have

- [ ] 命名迁移后的兼容窗口和后续包名策略有明确说明
- [ ] 样例矩阵覆盖不同题材，而不只是单一类型
- [ ] 发布前已有一份对外可读的 release summary
- [ ] 文档能区分“已实现能力”和“设计中能力”

## 4. Out Of Scope

以下内容不作为 `v1.0` 发布阻塞：

- 完整 Web UI
- 插图资产的完整可视化管理界面
- 生图 provider 的完全产品化收口
- 富文本编辑器
- 多人协作与权限系统
- 高自由度 workflow 编排
- 大规模模板市场
- 重型图像编辑能力

## 5. 验收矩阵

| 维度 | 必须回答的问题 | 通过标志 |
|------|----------------|----------|
| 协议 | 项目状态是否有明确真相源 | 文件协议稳定、状态域清晰 |
| 闭环 | 是否能从规划走到导出 | CLI 主链可跑通 |
| 长篇 | 是否不只是玩具 demo | 长篇样例验证通过 |
| 工作流 | 是否能显式记录决策 | `workflow` 可恢复、可导出 |
| 治理 | 是否适合外部发布 | 路线图、README、风险说明齐全 |
| 迁移 | Story Canvas 命名是否与现状一致 | 对外命名完成、CLI 兼容策略清楚 |

## 6. 样例项目建议口径

### 短篇回归样例

- 目标: 快速回归基础链路
- 应覆盖: `analyze` / `review chapter` / `review scene` / `context refresh` / `export`

### 长篇商业样例

- 目标: 验证卷级结构、章节字数目标、商业连载口径
- 应覆盖: 长篇 outline、workflow、review、导出

### 复杂长篇暗线样例

- 目标: 验证世界规则、长伏笔、角色动态状态等更重协议
- 优先参考: `demo-guixu`

## 7. 验证方式

### 基础验证

- [ ] `uv run story-canvas --help`
- [ ] `uv run python -m unittest discover -s tests`
- [ ] 核心命令在 `PYTHONPATH=src python -m story_canvas` 下可运行
- [ ] 兼容入口 `uv run story-harness --help` 与 `PYTHONPATH=src python -m story_harness_cli --help` 仍可运行

### 样例验证

- [ ] 短篇样例闭环跑通
- [ ] 长篇样例闭环跑通
- [ ] 复杂长篇样例至少完成协议加载、审查与上下文链路验证

### 文档验证

- [ ] `docs/roadmap.md` 与验收文档一致
- [ ] 样例相关说明文档同步
- [ ] 发布范围与非目标明确
- [ ] 提前并行开发的 UI / 生图能力未误写为 `v1.0` 阻塞项

## 8. 当前阻塞项

- [ ] 本地安装入口可能受镜像/证书影响
- [ ] 当前工作树存在并行改动，发布前需要收敛真实状态
- [ ] 部分 `v1.0` 能力仍处于“设计明确，代码未完全收口”的过渡阶段

## 9. 发布结论模板

### 可发布

满足条件：

- Must Have 全部通过
- 阻塞项已解除或降级为非阻塞风险

结论口径：

`v1.0` 已达到整体发布标准，可作为 Story Canvas 核心版本的第一次对外发布。

### 可发布但带风险

满足条件：

- Must Have 基本通过
- 存在非致命缺口，但已有明确 workaround

结论口径：

`v1.0` 可发布，但需在 release note 中明确列出已知限制与 workaround。

### 不可发布

满足条件：

- 任一核心协议域仍不稳定
- CLI 主闭环无法可靠验证
- 样例无法证明长篇与复杂项目可用

结论口径：

`v1.0` 暂不发布，继续收敛协议、闭环或样例验证。
